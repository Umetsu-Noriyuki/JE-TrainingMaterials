# IaCイベント通知構築<br>( EventBridge、SNS、Rule、Permission )

## 1. イベント通知構成とは？
イベント通知構成とは、**何かが起きたことをきっかけに、自動で処理や通知を行う仕組み**です。

例：
- 毎朝9時に定期バッチ起動
- ファイル保存時に通知送信
- 障害発生時に管理者へ連絡
- AWSイベント発生時に自動処理

AWSでは主に以下でイベントを利用します。

- **EventBridge**：イベント検知・スケジュール起動
- **Lambda**：自動処理
- **SNS**：通知配信
- **Permission**：サービス間の実行許可
- **CloudWatch Logs**：ログ確認
- **S3**：データ・ログ保存

```plaintext
9:00 ⇒ EventBridge/Rule
               ↓
             Lambda
               ↓
             SNS通知 ⇒ メール送信
```

👉 手動対応を減らし、業務自動化しやすくなります。

## 2. YAMLファイルの基本構造

イベント通知構成を IaC で管理する場合も、CloudFormation の基本構造は同じです。
1つのテンプレートの中に、EventBridge・Lambda・SNS など複数のリソースをまとめて記述します。

```plaintext
Root( テンプレート全体 )
├─ AWSTemplateFormatVersion
├─ Parameters( 入力値: 環境名/通知先メール/スケジュール設定 など )
├─ Resources( イベント通知構成の中心 )
│   ├─ AppEventRule( スケジュール/イベント条件 )
│   ├─ AppLambda( 実行処理 )
│   ├─ AppLambdaPermission( EventBridgeからの実行許可 )
│   ├─ AppLambdaLogGroup( Lambda実行ログ )
│   ├─ AppSnsTopic( 通知先グループ )
│   ├─ AppSnsSubscription( メール購読 )
│   └─ AppBucket( データ保存先 )
└─ Outputs( SNS Topic ARN/Lambda名 などの出力 )
```

### 各項目の役割
- **Parameters**  
  環境名、通知先メール、スケジュールなどの外部入力値を管理します。
- **Resources**  
  AppEventRule、AppLambda、AppSnsTopic などイベント通知構成の中心となる AWS リソースを定義します。
- **Outputs**  
  作成した SNS Topic ARN や Lambda 名など、後続確認で使う情報を出力します。

👉 この構造を理解しておくと、以降の EventBridge や SNS の IaC が読みやすくなります。

## 3. EventBridge / Rule のIaC構築
EventBridge は、イベントを受け取って別サービスへ連携する仕組みです。
起動条件は **Rule** で定義します。

### 3-1. Rule の主な種類
EventBridge の Rule は大きく以下の2種類があります。
| 種類 | 例 |
|---|---|
| 定時実行( スケジュール ) | 毎日 9:00 実行 |
| イベント検知( イベントトリガ ) | EC2停止の処理 / S3保存( ファイル到着 )時の処理 |

```plaintext
イベント( スケジュール/AWSイベント )
        ↓
   EventBridge/Rule
        ↓
      Lambda/SNS
```

👉 Rule は「いつ動かすか」を決め、`Targets` で呼び出し先を指定します。

### 3-2. 定時実行
指定した時刻や周期で、自動的に処理を実行します。
```yaml
Resources:
  AppEventRule:
    Type: AWS::Events::Rule
    Properties:
      # 実行条件を定義
      ScheduleExpression: cron(0 9 * * ? *)   # cron式
      # 呼び出し先を指定
      Targets:
        - Arn: !GetAtt AppLambda.Arn
          Id: AppLambdaTarget
```

👉 `ScheduleExpression` で「いつ動かすか」を定義し、`Targets` で呼び出し先を指定します。

#### cron式とは？
時刻や周期を細かく指定して処理を実行するスケジュール書式です。  
[AWS公式のcron式仕様 日本語版](https://docs.aws.amazon.com/ja_jp/systems-manager/latest/userguide/reference-cron-and-rate-expressions.html)  
　EventBridgeでも同様のcron式が使用されます  
　AWSのcron式仕様は複数サービスで共通のため、このドキュメントが参考になります  
( cronは元々Linuxのスケジューラで、その書式を利用しています )  

```plaintext
cron(0 9 * * ? *)
毎日 9:00 実行
```

#### 👉 スケジュールは Parameters 化することで、環境ごとに変更可能にできます
```yaml
Parameters:
  ScheduleExpression:
    Type: String
    Default: cron(0 9 * * ? *)

Resources:
  AppEventRule:
    Type: AWS::Events::Rule
    Properties:
      ScheduleExpression: !Ref ScheduleExpression
      Targets:
        - Arn: !GetAtt AppLambda.Arn
          Id: AppLambdaTarget
```

### 3-3. イベント検知
AWSサービスで発生したイベントをきっかけに処理を実行します。  
- EC2起動時
- S3保存時
- CodeBuild失敗時

以下はS3保存( ファイル到着 )時の例
```yaml
Resources:
  AppEventRule:
    Type: AWS::Events::Rule
    Properties:
      # 検知条件を定義
      EventPattern:
        source:
          - aws.s3
        detail-type:
          - Object Created
      # 呼び出し先を指定
      Targets:
        - Arn: !GetAtt AppLambda.Arn
          Id: AppLambdaTarget
```

👉 `EventPattern` で「何を契機に動かすか」を定義し、`Targets` で呼び出し先を指定します。

## 4. Lambda のIaC構築
Lambda は、EventBridge から呼び出されて実際の処理を担当するサービスです。

👉 EventBridge からの連携は、Rule の `Targets` に Lambda を指定して行います。

例：
- 定時バッチ処理
- 日報作成
- データ整形
- 通知本文の生成

```plaintext
EventBridge/Rule
       ↓
     Lambda
       ↓
  SNS/S3/Logs
```

### 最小構成例

```yaml
Resources:
  AppLambda:
    Type: AWS::Lambda::Function
    Properties:
      FunctionName: app-lambda
      Runtime: python3.12
      # Lambdaで最初に実行される関数( エントリポイント )
      Handler: index.lambda_handler
      # LambdaがAWSリソースへアクセスするための実行ロール
      Role: arn:aws:iam::123456789012:role/AppLambdaRole
      MemorySize: 256
      Timeout: 10
      Code:
        ZipFile: |
          def lambda_handler(event, context):
              return {"statusCode": 200}
```

👉 イベント通知構成では、Lambda が実際の処理本体になります。

## 5. S3 のIaC構築
CloudWatch Logs だけでなく、S3 にデータやログを保存する構成もよく使われます。  

### 利用例  
- Lambda の実行結果を JSON で保存
- 日報ファイルの保管
- 長期保管用ログの蓄積
- Athena / Glue で参照するための保存先

```plaintext
Lambda
  ↓
 S3
```

### S3 に保存するメリット
- 長期保存に向いている
- 他サービスと連携しやすい
- 保存データを後から再利用しやすい

### IaC 定義例
```yaml
Resources:
  AppBucket:
    Type: AWS::S3::Bucket
    Properties:
      BucketName: app-bucket-dev
```

👉 CloudWatch Logs は「調査」、S3 は「保存・再利用」で使い分けることが多いです。

## 6. SNS( 通知サービス )のIaC構築
SNS は、メッセージを複数宛先へ通知できるサービスです。  
イベント通知構成では、Lambda の処理結果や障害情報の配信に使われます。  

### 利用例
- メール通知
- 管理者への障害連絡
- 他システム連携
- Lambda からの通知配信

```plaintext
Lambda
  ↓
  SNS
  ├─ メール通知
  ├─ Lambda起動
  └─ 他システム連携
```

### Topic 作成例
```yaml
Resources:
  AppSnsTopic:
    # 通知を配信するためのグループ( Topic )
    Type: AWS::SNS::Topic
```

### 購読設定例
```yaml
Resources:
  AppSnsSubscription:
    # 通知の受信先( メールなど )を登録
    Type: AWS::SNS::Subscription
    Properties:
      Protocol: email
      Endpoint: sample@example.com
      TopicArn: !Ref AppSnsTopic
```

👉 SNS( **AppSnsTopic / AppSnsSubscription** )は「誰に知らせるか」を担うリソースです。

## 7. Permission( 実行許可 )のIaC構築
AWSサービス同士が連携するには、実行許可が必要です。  
イベント通知構成では、**作成したあとに Permission 不足で動かない**ケースがよくあります。

例：  
- EventBridge → Lambda実行
- SNS → Lambda実行

```plaintext
EventBridge/Rule
       ↓ 許可必要
     Lambda
```

### Lambda 許可例
```yaml
Resources:
  AppLambdaPermission:
    Type: AWS::Lambda::Permission
    Properties:
      FunctionName: !Ref AppLambda
      Action: lambda:InvokeFunction
      # Lambdaを呼び出すサービス( ここではEventBridge )
      Principal: events.amazonaws.com
      # 呼び出し元リソース( EventBridge Rule )の指定
      SourceArn: !GetAtt AppEventRule.Arn
```

👉 Lambda や EventBridge のリソースは作成できても、Permission 不足では動作しません。

## 8. イベント通知で利用するParametersのIaC設計
Parameters は、テンプレート内で後から変更できる入力項目です。
同じYAMLでも、値を変えるだけで別環境や別設定に使い回せます。

### よく使うパラメータ
- 環境名( dev/stag/prod )
- 通知先メールアドレス
- スケジュール式
- S3 バケット名
- Lambda 関数名の接頭辞

```yaml
Parameters:
  # 環境名( dev/stag/prodなどの識別子 )
  EnvName:
    Type: String
    Default: dev

  # 通知先メールアドレス
  NotifyEmail:
    Type: String

  # 実行スケジュール( cron など )を指定
  ScheduleExpression:
    Type: String
    Default: cron(0 9 * * ? *)
```

👉 イベント通知構成では、通知先やスケジュールを Parameters 化しておくと管理しやすくなります。

## 11. よくある失敗例

| 失敗 | 原因 |
|---|---|
| 定時起動しない | cron設定ミス |
| Lambda呼ばれない | Permission不足 |
| 通知が来ない | SNS購読未確認 |
| ログが見えない | LogGroup / 権限設定不足 |
| S3保存されない | バケット名・権限設定ミス |
| dev/prod誤通知 | Topic分離不足<br>( 開発環境と本番環境で同じSNS Topicを使っていると、<br>　意図しない通知が混在する ) |

## 12. まとめ
- EventBridge / Rule は起動条件を定義する
- Lambda は実処理を担当する
- CloudWatch Logs は実行結果の確認に使う
- S3 は保存先として使える
- SNS は通知配信を担当する
- Permission はサービス間実行許可を担う
- Parameters 化で環境差分を管理しやすくなる

👉 イベント通知構成は **業務自動化・障害検知・定時処理** に強い仕組みです。

## 13. 重要用語説明

| 用語 | 説明 |
|---|---|
| EventBridge( イベント連携 ) | イベント発生や定時実行を起点に処理を連携するサービス。 |
| Rule( 実行条件 ) | EventBridgeでイベント条件を定義する設定。 |
| Lambda( ラムダ ) | サーバー管理不要でプログラムを実行できるサービス。 |
| SNS( 通知サービス )<br>Simple Notification Service | メール等へ通知配信できるサービス。 |
| Permission( 実行許可 ) | AWSサービス間の実行許可設定。 |
| CloudWatch Logs( ログ保存機能 ) | Lambda のログを保存・確認するサービス。 |
| S3( オブジェクトストレージ ) | データやログを保存するストレージサービス。 |
| cron expression( 定時実行式 ) | 時刻や周期を指定する定時実行書式。 |
| Deploy( デプロイ ) | コードや設定をAWS環境へ適用すること。 |
| Event Driven( イベント駆動 )<br>Event Driven Architecture | イベント発生を起点に処理する設計方式。 |
| Schedule( 定時実行 ) | 指定時刻や周期で自動実行する仕組み。 |
| Notification( 通知 ) | 利用者や管理者へ知らせる処理。 |
| Topic( 通知先グループ ) | SNSで配信先をまとめる単位。 |
| Subscription( 購読設定 ) | SNS通知の受信先登録設定。 |
