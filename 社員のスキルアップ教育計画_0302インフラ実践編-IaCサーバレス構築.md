# IaCサーバレス構築<br>( Lambda、API Gateway、IAM Role、CloudWatch Logs )

## 1. サーバレス構成とは？

サーバレス構成とは、サーバー台数やOS管理を意識せず、  
**必要な処理だけを実行するクラウド構成**です。

AWSでは主に以下を組み合わせます。

- **Lambda**：プログラム実行
- **API Gateway**：HTTP API公開
- **IAM Role**：権限付与
- **CloudWatch Logs**：ログ確認

```plaintext
利用者
  ↓ HTTPアクセス
API Gateway
  ↓
Lambda
  ↓
CloudWatch Logs
```

👉 インフラ管理負荷を下げ、開発スピードを上げやすい構成です。

## 2. YAMLファイルの基本構造
サーバレス構成を IaC で管理する場合も、CloudFormation の基本構造は同じです。  
1つのテンプレートの中に、Lambda・API Gateway・IAM Role など複数のリソースをまとめて記述します。

```plaintext
Root
├─ AWSTemplateFormatVersion
├─ Parameters（環境差分の入力値）
├─ Resources（サーバレス構成の中心）
│   ├─ AppLambda（Lambda関数）
|   ├─ AppLambdaPermission（API Gatewayからの実行許可）
│   ├─ AppLambdaRole（IAM Role）
│   ├─ AppApi（API Gateway）
│   └─ AppLambdaLogGroup（ログ設定）
└─ Outputs（API URL などの出力）
```

### 各項目の役割
 - Parameters  
 dev / prod の切り替えや Lambda の設定値を外部入力として管理
 - Resources  
 AppLambda、AppApi、AppLambdaRole などサーバレス構成の中心となるリソースを定義
 - Outputs  
 作成した API の URL や Lambda ARN を外部へ出力する際に使用  

👉 この構造を理解しておくと、以降の AppLambda や AppApi の IaC が読みやすくなります。

## 3. Lambda を IaC で定義する
ここでは、Lambda 関数の設定（関数名・ランタイム・メモリ・タイムアウト・実行ロールなど）を
CloudFormation でどのように定義するかを扱います。  
アプリケーションの実行環境をコードとして管理することで、環境差分の統一や再現性が高まります。

### 例：最小構成

```yaml
Resources:
  AppLambda:
    Type: AWS::Lambda::Function
    Properties:
      FunctionName: app-lambda
      Runtime: python3.12
      Handler: index.lambda_handler
      Role: arn:aws:iam::123456789012:role/AppLambdaRole
      MemorySize: 256
      Timeout: 10
      Code:
        ZipFile: |
          def lambda_handler(event, context):
              return {"statusCode": 200}
```

## 4. API Gateway を IaC で定義する

Lambda を外部公開する際に利用するサービスが **API Gateway** です。  
この章では、API を正しく動作させるために必要となる **API Gateway の構成要素** を IaC（CloudFormation）でどのように定義するかを解説します。  

API Gateway は RestApi を作成するだけでは動作せず、
**パス（Resource）・メソッド（Method）・Lambda との紐付け（Integration）・呼び出し許可（Permission）**
といった複数の設定が揃って初めて API が機能します。  

この章では、これらの構成要素がどのように連携し、
どのように IaC で定義すべきかを順を追って説明します。

```plaintext
ブラウザ
 ↓
API Gateway
 ↓
Lambda
```

### Lambdaで実行するコード（Lambda関数）の利用例

* Web API
* 社内ツールAPI
* LINE Bot連携
* 外部システムWebhook

### 定義イメージ

```yaml
Resources:
  AppApi:
    Type: AWS::ApiGateway::RestApi
    Properties:
      Name: app-api
```

👉 ただし RestApi を作っただけでは API は動作しません。

### API を動作させるために必要な構成

- Resource（パス）：例 /hello
- Method（GET/POST など）
- Integration（Lambda との紐付け）
- Lambda Permission（API → Lambda 呼び出し権限）
これらが揃って初めて API が正常に動作します。

## 5. IAM Role

サーバレス構成では、**Lambda がどの AWS サービスにアクセスできるか** を明確に制御する必要があります。  
そのために利用するのが **IAM Role（実行ロール）** です。  
この章では、Lambda が実行時に利用する IAM Role を IaC でどのように定義し、
どのような権限を付与すべきかを解説します。  
特に、CloudWatch Logs・S3・DynamoDB など、
サーバレス構成で頻繁に利用されるサービスへのアクセス権限を中心に、
**最小権限の考え方** と 実務でよく使うポリシー を紹介します。  

例：

```yaml
Resources:
  AppLambdaRole:
    Type: AWS::IAM::Role
    Properties:
      AssumeRolePolicyDocument:
        Version: "2012-10-17"
        Statement:
          - Effect: Allow
            Principal:
              Service: lambda.amazonaws.com
            Action: sts:AssumeRole
```

### よく使う権限
 - CloudWatch Logs 出力
 - S3 読み書き
 - DynamoDB 読み書き

👉 サーバレスでは「Lambda が何をできるか」を Role で明確にすることが重要です。

## 6. ログ管理とサービス間連携の基礎
サーバレス構成では、**ログの扱い** と **サービス間の呼び出し許可（Permission）** が
正しく設定されていないと、原因調査が困難になったり、サービスが動作しないといった問題が発生します。  
この章では、以下の 3 点を中心に、実務で必須となる設定とその理由を解説します。

 - 6-1. CloudWatch Logs の確認と Retention 設定  
 Lambda のログがどのように出力されるか、なぜ保持期間を IaC で管理すべきなのかを説明します。
 - 6-2. Permission（呼び出し許可）の仕組み  
 API Gateway や EventBridge など、AWS サービス同士が連携する際に必要となる  
*AWS::Lambda::Permission* の役割と、設定漏れで起きる典型的な失敗を解説します。
 - 6-3. S3 へのログ出力  
 CloudWatch Logs だけでなく、長期保管や分析用途で S3 にログを保存する構成について紹介します。

 👉 この章を通じて、「ログの管理」と「呼び出し許可」 というサーバレス運用の基礎を体系的に理解できます。

## 6-1. CloudWatch Logs を確認する

Lambda の実行ログは CloudWatch Logs に出力されます。
実行時に LogGroup が自動作成されますが、保持期間はデフォルトで無期限のため、
IaC で明示的に Retention を設定するのが実務では一般的です。

例：

* print の内容
* エラー原因
* 実行時間
* メモリ使用量

### LogGroup を IaC で管理する例

```yaml
Resources:
  AppLambdaLogGroup:
    Type: AWS::Logs::LogGroup
    Properties:
      LogGroupName: /aws/lambda/app-lambda
      RetentionInDays: 14
```

👉 無期限保持はコスト増につながるため、Retention 設定は実務で必須です。

### 無期限保持がコスト増につながる理由
CloudWatch Logs は **保存したログ量（GB）× 保存期間（日数）** に応じて料金が発生します。  
保持期間が無期限だと、**古いログが増え続け、保存コストが積み上がる**ため、長期運用では確実にコスト増になります。

### Retention 設定とは何か
**Retention（保持期間）とは、CloudWatch Logs のログを自動削除するまでの日数を指定する設定**です。

例：
 - 14日
 - 30日
 - 90日
 - 1年
 - 無期限（デフォルト）  

 実務では、**「調査に必要な期間だけ残し、それ以外は自動削除する」**という運用が一般的です。

## 6-2. Permission（呼び出し許可）を IaC で定義する
AWS のサービス同士が連携する際には、**「呼び出される側のサービスが、呼び出し元を許可する」** という設定が必要です。  
Lambda の場合、
 - API Gateway → Lambda
 - EventBridge → Lambda
 - SNS → Lambda

など、どのサービスが Lambda を呼び出す場合でも、**AWS::Lambda::Permission** を設定しないと動作しません。

### Permission が必要になる理由
 - IAM Role は **Lambda が外部サービスへアクセスする権限**  
 - Permission は **外部サービスが Lambda を呼び出す権限**  

👉 **方向が逆であり、別の仕組み**です。

### Permission がないとどうなるか
イベント通知構成では特に、
 - リソースは作成できる
 - EventBridge ルールも作成できる
 - SNS トピックも作成できる
 - Lambda も作成できる

にもかかわらず…  
**通知が来ても Lambda が動かない（Permission 不足）**  
という失敗が非常に多いです。

### 典型例
```plaintext
EventBridge/Rule
       ↓（呼び出し許可が必要）
     Lambda
```

### API Gateway → Lambda の Permission 例
```yaml
Resources:
  AppLambdaPermission:
    Type: AWS::Lambda::Permission
    Properties:
      FunctionName: !Ref AppLambda
      Action: lambda:InvokeFunction
      Principal: apigateway.amazonaws.com
      SourceArn: !Sub arn:aws:execute-api:${AWS::Region}:${AWS::AccountId}:${AppApi}/*/GET/hello
```

### EventBridge → Lambda の Permission 例
```yaml
Resources:
  AppLambdaPermission:
    Type: AWS::Lambda::Permission
    Properties:
      FunctionName: !Ref AppLambda
      Action: lambda:InvokeFunction
      Principal: events.amazonaws.com
      SourceArn: !GetAtt AppEventRule.Arn
```

👉 EventBridge や SNS のリソースは作成できても、Permission がないと Lambda は実行されません。

## 6-3.S3 にログを出力するケース
CloudWatch Logs だけでなく、S3 にログを保存する構成もよく使われます。

### 例：
 - Lambda の実行結果を JSON で S3 に保存
 - アプリケーションログを長期保管
 - 分析基盤（Athena / Glue）で参照するためのログ蓄積

 ```plaintext
 Lambda
   ↓（ログ保存）
 S3（長期保管・分析用）
```
### S3 にログを保存するメリット
 - 長期保存が安価（CloudWatch Logs よりコスト効率が良い）
 - Athena で直接クエリできる
 - 外部システムとの連携がしやすい

 ### IaC での S3 バケット定義例

```yaml
Resources:
  AppBucket:
    Type: AWS::S3::Bucket
    Properties:
      BucketName: app-bucket-dev
```
👉 CloudWatch Logs は「短期の調査」、S3 は「長期保管・分析」という使い分けが一般的です。

## 7. サーバレス構成で使う Parameters

サーバレス構成では、**環境（dev/prod）**ごとに異なる値をテンプレートへ渡すために **Parameters を利用します**。  
メモリサイズやタイムアウト、ステージ名など、環境差分が出る設定をコードに直接書かず、
**Parameters として外部から与えることでテンプレートの再利用性が高まります**。  
この章では、サーバレス構成でよく使う代表的なパラメータと、その定義方法を紹介します。

### よく使うパラメータ
 - Lambda メモリ（dev=256 / prod=512）
 - タイムアウト（dev=5秒 / prod=15秒）
 - API Gateway のステージ名（dev / prod）
 - Lambda 関数名の接頭辞

### サーバレス構成の定義例

```yaml
Parameters:
  EnvName:
    Type: String
    Default: dev
```

## 8. サーバレス構成の実行コマンド

### デプロイ

```bash
aws cloudformation deploy \
--template-file template.yaml \
--stack-name serverless-stack \
--capabilities CAPABILITY_NAMED_IAM \
--parameter-overrides EnvName=dev
```

### 削除

```bash
aws cloudformation delete-stack \
--stack-name serverless-stack
```

## 10. よくある失敗例

| 失敗 | 原因 |
| ---------- | --------------- |
| Lambda実行失敗 | IAM権限不足 |
| タイムアウト | デフォルト3秒のまま |
| VPC内Lambdaが遅い | ENI（Elastic Network Interface：ネットワークインターフェース）作成待ち |
| API が呼べない | Method / Integration 設定漏れ |
| 外部APIに接続不可 | SG の egress（送信方向）制限により外部通信がブロックされている |
| ログが見れない | Logs権限不足 |
| dev/prod混在 | ステージや環境変数の設定不足により、異なる環境の値が混在している |

## 11. まとめ

* AppLambda は処理実行担当
* API Gateway は公開入口
* AppLambdaRole は権限管理
* CloudWatch Logs は調査用
* サーバレスは IaC にすることで、環境差分管理が容易になる

👉 サーバレス構成は **小さく始めて拡張しやすい実務向け構成** です。

## 12. 重要用語説明

| 用語 | 説明 |
|---|---|
| Lambda( 関数実行サービス ) | サーバー管理不要でプログラムを実行できるサービス。 |
| IAM Role（権限付与） | AWS リソースに実行権限を付与する仕組み。Lambda が他サービスにアクセスする際に必要。 |
| API Gateway( API管理 ) | API を公開・管理するサービス |
| CloudWatch Logs( ログ保存機能 ) | Lambda のログを保存・検索するサービス |
| S3（オブジェクトストレージ） | ファイルやログを保存するストレージサービス。<br>Lambda の実行ログやアプリケーションログの長期保存にも利用される。|
| Trigger( 起動条件 ) | Lambda を起動するイベント |
| Timeout( 制限時間 ) | Lambda の最大実行時間 |
| Endpoint( 接続先URL ) | API の URL |
| Integration（統合設定） | API Gateway と Lambda の接続設定 |
| Lambda Permission（呼び出し権限） | API Gateway が Lambda を呼ぶための権限 |
| DynamoDB（NoSQLデータベース） | スキーマレスで高速なデータベース。Lambda からのデータ保存に利用される。 |
| Stage（ステージ） | API Gateway の環境（dev / prod） |
