# IaCサーバレス構築<br>( Lambda、API Gateway、IAM Role、CloudWatch Logs )

## 1. サーバレス構成とは？

サーバレス構成とは、サーバー台数やOS管理を意識せず、  
**必要な処理だけを実行するクラウド構成**です。

AWSでは主に以下を組み合わせます。

- **Lambda**：プログラム実行
- **API Gateway**：HTTP API公開
- **IAM Role**：権限付与
- **CloudWatch Logs**：ログ確認

```plaintext
利用者
  ↓ HTTPアクセス
API Gateway
  ↓
<<<<<<< HEAD

=======
Lambda
>>>>>>> main
  ↓
CloudWatch Logs
```

👉 インフラ管理負荷を下げ、開発スピードを上げやすい構成です。

## 2. サーバレスを IaC で構築するメリット

サーバレスでは特に以下の効果が大きくなります。

 - API / Lambda の構成をコード化できる
 - dev / prod のステージを簡単に複製できる
 - Lambda のメモリ・タイムアウトなどをコードで管理できる
 - IAM Role をテンプレートで統一できる

👉 サーバレスは IaC と相性が非常に良い構成です。

## 3. Lambda を IaC で定義する

Lambda はコードを実行するサービスです。  
Lambda は実行時に CloudWatch Logs の LogGroup を自動作成しますが、保持期間はデフォルトで無期限のため、IaC で明示的に設定するのが実務では一般的です。

## 2. IaCで構築する意義

管理画面で手動作成も可能ですが、実務ではIaC化が重要です。

## 6-1. CloudWatch Logs を確認する

Lambda の実行ログは CloudWatch Logs に出力されます。

例：

* print の内容
* エラー原因
* 実行時間
* メモリ使用量

### LogGroup を IaC で管理する例

```yaml
Resources:
  AppLambdaLogGroup:
    Type: AWS::Logs::LogGroup
    Properties:
      LogGroupName: /aws/lambda/app-lambda
      RetentionInDays: 14
```

👉 無期限保持はコスト増につながるため、Retention 設定は実務で必須です。

## 6-2.S3 にログを出力するケース
CloudWatch Logs だけでなく、S3 にログを保存する構成もよく使われます。
### 例：
 - Lambda の実行結果を JSON で S3 に保存
 - アプリケーションログを長期保管
 - 分析基盤（Athena / Glue）で参照するためのログ蓄積

 ```plaintext
 AppLambda
   ↓（ログ保存）
     S3（長期保管・分析用）
```
### S3 にログを保存するメリット
 - 長期保存が安価（CloudWatch Logs よりコスト効率が良い）
 - Athena で直接クエリできる
 - 外部システムとの連携がしやすい

 ### IaC での S3 バケット定義例

```yaml
Resources:
  AppBucket:
    Type: AWS::S3::Bucket
    Properties:
      BucketName: app-bucket-dev
```
👉 CloudWatch Logs は「短期の調査」、S3 は「長期保管・分析」という使い分けが一般的です。

## 10. よくある失敗例

| 失敗 | 原因 |
| ---------- | --------------- |
| Lambda実行失敗 | IAM権限不足 |
| タイムアウト | デフォルト3秒のまま |
| VPC内Lambdaが遅い | ENI作成待ち |
| API が呼べない | Method / Integration 設定漏れ |
| 外部APIに接続不可 | SGの egress 制限 |
| ログが見れない | Logs権限不足 |
| dev/prod混在 | ステージ管理不足 |
