# IaCサーバレス構築<br>( Lambda、API Gateway、IAM Role、CloudWatch Logs )

## 1. サーバレス構成とは？

サーバレス構成とは、サーバー台数やOS管理を意識せず、  
**必要な処理だけを実行するクラウド構成**です。

AWSでは主に以下を組み合わせます。

- **Lambda**：プログラム実行
- **API Gateway**：HTTP API公開
- **IAM Role**：権限付与
- **CloudWatch Logs**：ログ確認

```plaintext
利用者
  ↓ HTTPアクセス
API Gateway
  ↓
AppLambda
  ↓
CloudWatch Logs
```

👉 インフラ管理負荷を下げ、開発スピードを上げやすい構成です。

## 2. YAMLファイルの基本構造
サーバレス構成を IaC で管理する場合も、CloudFormation の基本構造は同じです。  
1つのテンプレートの中に、Lambda・API Gateway・IAM Role など複数のリソースをまとめて記述します。

```plaintext
Root
├─ AWSTemplateFormatVersion
├─ Parameters（環境差分の入力値）
├─ Resources（サーバレス構成の中心）
│   ├─ AppLambda（Lambda関数）
│   ├─ AppLambdaRole（IAM Role）
│   ├─ AppApi（API Gateway）
│   └─ AppLambdaLogGroup（ログ設定）
└─ Outputs（API URL などの出力）
```

### 各項目の役割
 - Parameters  
 dev / prod の切り替えや Lambda の設定値を外部入力として管理
 - Resources  
 AppLambda、AppApi、AppLambdaRole などサーバレス構成の中心となるリソースを定義
 - Outputs  
 作成した API の URL や Lambda ARN を外部へ出力する際に使用  

👉 この構造を理解しておくと、以降の AppLambda や AppApi の IaC が読みやすくなります。

## 3. Lambda を IaC で定義する

Lambda はコードを実行するサービスです。  
Lambda は実行時に CloudWatch Logs の LogGroup を自動作成しますが、保持期間はデフォルトで無期限のため、IaC で明示的に設定するのが実務では一般的です。


### 例：最小構成

```yaml
Resources:
  AppLambda:
    Type: AWS::Lambda::Function
    Properties:
      FunctionName: app-lambda
      Runtime: python3.12
      Handler: index.lambda_handler
      Role: arn:aws:iam::123456789012:role/AppLambdaRole
      MemorySize: 256
      Timeout: 10
      Code:
        ZipFile: |
          def lambda_handler(event, context):
              return {"statusCode": 200}
```

## 4. API Gateway を IaC で定義する

Lambda を外部公開する際に利用します。

```plaintext
ブラウザ
 ↓
API Gateway
 ↓
AppLambda
```

### 利用例

* Web API
* 社内ツールAPI
* LINE Bot連携
* 外部システムWebhook

### 定義イメージ

```yaml
Resources:
  AppApi:
    Type: AWS::ApiGateway::RestApi
    Properties:
      Name: app-api
```

👉 ただし RestApi を作っただけでは API は動作しません。

### API を動作させるために必要な構成

- Resource（パス）：例 /hello
- Method（GET/POST など）
- Integration（Lambda との紐付け）
- Lambda Permission（API → Lambda 呼び出し権限）
これらが揃って初めて API が正常に動作します。

## 5. AppLambdaRole

IIAM Role は AppLambda が他サービスへアクセスするための権限です。
例：

* CloudWatch Logsへ出力
* S3へ保存
* DynamoDB参照

```yaml
Resources:
  AppLambdaRole:
    Type: AWS::IAM::Role
    Properties:
      AssumeRolePolicyDocument:
        Version: "2012-10-17"
        Statement:
          - Effect: Allow
            Principal:
              Service: lambda.amazonaws.com
            Action: sts:AssumeRole
```

### よく使う権限
 - CloudWatch Logs 出力
 - S3 読み書き
 - DynamoDB 読み書き

👉 サーバレスでは「Lambda が何をできるか」を Role で明確にすることが重要です。

## 6-1. CloudWatch Logs を確認する

Lambda の実行ログは CloudWatch Logs に出力されます。

例：

* print の内容
* エラー原因
* 実行時間
* メモリ使用量

### LogGroup を IaC で管理する例

```yaml
Resources:
  AppLambdaLogGroup:
    Type: AWS::Logs::LogGroup
    Properties:
      LogGroupName: /aws/lambda/app-lambda
      RetentionInDays: 14
```

👉 無期限保持はコスト増につながるため、Retention 設定は実務で必須です。

## 6-2.S3 にログを出力するケース
CloudWatch Logs だけでなく、S3 にログを保存する構成もよく使われます。
### 例：
 - Lambda の実行結果を JSON で S3 に保存
 - アプリケーションログを長期保管
 - 分析基盤（Athena / Glue）で参照するためのログ蓄積

 ```plaintext
 AppLambda
   ↓（ログ保存）
     S3（長期保管・分析用）
```
### S3 にログを保存するメリット
 - 長期保存が安価（CloudWatch Logs よりコスト効率が良い）
 - Athena で直接クエリできる
 - 外部システムとの連携がしやすい

 ### IaC での S3 バケット定義例

```yaml
Resources:
  AppBucket:
    Type: AWS::S3::Bucket
    Properties:
      BucketName: app-bucket-dev
```
👉 CloudWatch Logs は「短期の調査」、S3 は「長期保管・分析」という使い分けが一般的です。

## 7. サーバレス構成で使う Parameters

ここでは サーバレスでよく使う差分 に絞ります。

### よく使う差分
 - Lambda メモリ（dev=256 / prod=512）
 - タイムアウト（dev=5秒 / prod=15秒）
 - API Gateway のステージ名（dev / prod）
 - Lambda 関数名の接頭辞

```yaml
Parameters:
  EnvName:
    Type: String
    Default: dev
```

## 8. サーバレス構成の実行コマンド

### デプロイ

```bash
aws cloudformation deploy \
--template-file template.yaml \
--stack-name serverless-stack \
--capabilities CAPABILITY_NAMED_IAM \
--parameter-overrides EnvName=dev
```

### 削除

```bash
aws cloudformation delete-stack \
--stack-name serverless-stack
```

## 9. よくあるサーバレス構成

```plaintext
API利用者
   ↓
API Gateway
   ↓
AppLambda
   ├─ S3保存
   ├─ DynamoDB保存
   └─ CloudWatch Logs
```
API Gateway が Lambda を呼び出すには、AppLambda 側に `AWS::Lambda::Permission` が必要 です。

### CloudFormation テンプレートの全体構成（サーバレス構成）

```plaintext
Root
├─ AWSTemplateFormatVersion
├─ Parameters（環境差分の入力値）
├─ Resources（サーバレス構成の中心）
│   ├─ AppLambda（Lambda関数）
│   ├─ AppLambdaRole（IAM Role）
│   ├─ AppApi（API Gateway）
│   ├─ AppBucket（S3）
│   ├─ AppTable（DynamoDB）
│   └─ AppLambdaLogGroup（ログ設定）
└─ Outputs（API URL などの出力）
```
👉 この構成を理解しておくと、サーバレス構成の IaC 全体像がつかみやすくなります。

## 10. よくある失敗例

| 失敗 | 原因 |
| ---------- | --------------- |
| Lambda実行失敗 | IAM権限不足 |
| タイムアウト | デフォルト3秒のまま |
| VPC内Lambdaが遅い | ENI作成待ち |
| API が呼べない | Method / Integration 設定漏れ |
| 外部APIに接続不可 | SGの egress 制限 |
| ログが見れない | Logs権限不足 |
| dev/prod混在 | ステージ管理不足 |

## 11. まとめ

* AppLambda は処理実行担当
* API Gateway は公開入口
* AppLambdaRole は権限管理
* CloudWatch Logs は調査用
* サーバレスは IaC と相性が良く、環境差分管理が容易

👉 サーバレス構成は **小さく始めて拡張しやすい実務向け構成** です。

## 12. 重要用語説明

| 用語 | 説明 |
|---|---|
| Lambda( 関数実行サービス ) | サーバー管理不要でプログラムを実行できるサービス。 |
| IAM Role（権限付与） | AWS リソースに実行権限を付与する仕組み。Lambda が他サービスにアクセスする際に必要。 |
| API Gateway( API管理 ) | API を公開・管理するサービス |
| CloudWatch Logs( ログ保存機能 ) | Lambda のログを保存・検索するサービス |
| S3（オブジェクトストレージ） | ファイルやログを保存するストレージサービス。<br>Lambda の実行ログやアプリケーションログの長期保存にも利用される。}
| Trigger( 起動条件 ) | Lambda を起動するイベント |
| Timeout( 制限時間 ) | Lambda の最大実行時間 |
| Endpoint( 接続先URL ) | API の URL |
| Integration（統合設定） | API Gateway と Lambda の接続設定 |
| Lambda Permission（呼び出し権限） | API Gateway が Lambda を呼ぶための権限 |
| DynamoDB（NoSQLデータベース） | スキーマレスで高速なデータベース。Lambda からのデータ保存に利用される。 |
| Stage（ステージ） | API Gateway の環境（dev / prod） |
