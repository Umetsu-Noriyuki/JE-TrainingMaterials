# IaCトラブルシュート<br>( CREATE_FAILED、ROLLBACK、IAM、Dependency、Drift )

## 1. IaCトラブルシュートとは？
CloudFormation は便利ですが、  
実務では **作成失敗・更新失敗・削除失敗** が発生することがあります。

例えば：
- Stack 作成が途中で止まる
- 更新時にロールバックされる
- 削除できない
- 権限不足で失敗する
- 手作業変更で差異が出る

👉 エラー時に落ち着いて原因を確認する力が重要です。

## 2. まず確認する場所
トラブル時は **AWSコンソールの CloudFormation** で Events を確認します。

```plaintext
CloudFormation > 対象Stack > Events タブ
```
※ 対象Stackは、今回操作しているStack名から特定して選択します

[ 確認ポイント ]  
- 失敗したリソース名
- Status reason( 失敗理由 )
- エラーメッセージ
- どの処理で止まったか

👉 最初に Events を見る習慣が重要です。

## 3. おもなトラブル
CloudFormation の失敗原因はさまざまですが、  
まずは発生しやすいパターンを押さえることが重要です。  

ここでは、作成・更新・権限・削除・構成差異の代表例を確認します。

### 3-1. CREATE_FAILED
[ 例 ] 作成時に失敗した状態です。  
```plaintext
CREATE_FAILED
Bucket already exists
```

[ よくある原因 ]
* リソース名重複
* IAM 権限不足
* 必須パラメータ不足
* テンプレート構文ミス

👉 原因に応じて設定、権限、パラメータを見直します。

### 3-2. UPDATE_ROLLBACK_COMPLETE
[ 例 ] 更新失敗後、自動的に元へ戻された状態です。  
```plaintext
UPDATE_ROLLBACK_COMPLETE
Resource update failed
```
これは「成功した」という意味ではなく、
**更新に失敗したため、元の状態へ戻された**という意味です。

```plaintext
変更適用
 ↓
失敗
 ↓
自動ロールバック
```

[ 主な原因 ]
- 変更内容の不整合
- 削除できない依存関係
- 一部サービス制約

👉 変更差分を見直して再実行します。

### 3-3. IAM権限不足
[ 例 ] 権限不足により操作が拒否された状態です。  
```plaintext
AccessDenied
User is not authorized to perform: lambda:CreateFunction
```
※ IAMは「実行者側」と「リソース側(Role)」の2系統ある点に注意

[ 確認点 ]
- 実行ユーザー権限
- CloudFormation 実行Role
- 作成対象サービス権限

👉 IAM不足は非常に多い失敗原因です。

### 3-4 削除できないケース
[ 例 ] リソースの状態や依存関係により削除できない状態です。  
```plaintext
DELETE_FAILED
The bucket you tried to delete is not empty
```

[ よくある例 ]
- S3 バケット内にファイルが残っている
- ENI が残っている
- Security Group が参照中
- RDS 削除保護が有効

```plaintext
delete-stack
 ↓
DELETE_FAILED
```

👉 依存先を解除してから再実行します。

### 3-5. Drift( 構成差異 )
[ 例 ] テンプレートと実環境の設定が一致していない状態です。  

```plaintext
テンプレート:
  EC2 t3.micro

実環境:
  手作業で t3.large
```

( Driftはエラーではなく、構成差異として検出される )  
手作業変更( 運用のズレ )により、構成差異( Drift )が発生することが多い  

※ すべての差異が検出できるわけではない点に注意

[ 例 ]
```plaintext
テンプレート:
  EC2 t3.micro

実環境:
  手作業で t3.large
```

[ 問題点 ]
- 再デプロイで戻る
- 原因不明障害になる
- 管理不能になる

👉 AWS Console 直接変更を避け、コード修正を基本にします。

## 4. 調査の基本手順
```plaintext
1. [ Events確認 ]
       ↓
2. [ 失敗リソース確認 ]
       ↓
3. [ Status reason / エラー文確認 ]
       ↓
4. [ IAM / 依存関係確認 ]
       ↓
5. [ テンプレート修正 ]
       ↓
6. [ 再実行 ]
```

👉 感覚ではなく、順番に確認することが重要です。

## 5. まとめ
- **エラー確認**
  - Events を確認する
  - Status reason( エラー文 ) を読む
  - 失敗したリソースを特定する

- **現象別の確認ポイント**
  - 作成失敗( CREATE_FAILED )  
    　⇒ 設定, 権限, パラメータ, 名前重複 を確認する
  - 更新失敗( UPDATE_ROLLBACK_COMPLETE )  
    　 ⇒ 差分, 依存関係 を確認する
  - 権限エラー  
    　⇒ IAM( 実行者とRole ) を確認する
  - 削除失敗  
    　⇒ 依存関係, リソース状態 を確認する
  - 構成差異( Drift )  
    　⇒ 手作業変更の有無 を確認する

👉 IaC は「作る力」と同じくらい「直す力」が重要です。

## 6. 重要用語説明
| 用語 | 説明 |
|---|---|
| Drift( 構成差異 )<br>Configuration Drift | コード定義と実環境設定がズレること。 |
