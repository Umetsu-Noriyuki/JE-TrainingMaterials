# ネットワーク・サーバー基盤<br>( VPC、EC2、ECS+EC2、ECS+Fargate、Security Group )

## 1. 基盤を支えるネットワーク - VPC( Virtual Private Cloud )とは？
AWSでのシステムの「家」を作るイメージです。  
**VPC( Virtual Private Cloud )** は、クラウド上に自分専用のネットワーク空間を作り、  
他のユーザーと分離した安全な環境を構築します。

- **目的**：安全にリソースを配置し、通信経路を管理  
- **特徴**：プライベートIPアドレス空間を自由に割り当て可能  
- **例**：オフィスの社内LANをクラウド上に作るイメージ

```plaintext
インターネット
    │
  VPC( プライベートネットワーク )
    ├── 各種サービス
    └── 各種サービス
```
※具体的なサービス例は後の章で詳細に説明します。

## 2. Subnet( サブネット ) - Public / Private の配置設計
VPCの中を、用途ごとに分割したネットワーク区画です。  
同じVPC内でも、公開用と非公開用で配置場所を分けることで、安全で管理しやすい構成にできます。  

### 主な種類
- **Public Subnet**  
  インターネットと接続できる区画です。  
  踏み台サーバ、公開Webサーバなどを配置します。

- **Private Subnet**  
  インターネットから直接アクセスさせない区画です。  
  内部サーバ、データベースなどを配置します。

### 構成イメージ
```plaintext
インターネット
    │
 VPC( プライベートネットワーク )
    ├── Public Subnet
    │      └── EC2( 踏み台サーバ )
    └── Private Subnet
           └── EC2( 内部サーバ )
```

👉 **どのサーバをどこに置くか( Subnet設計 ) が、セキュリティと運用性に大きく影響します。**

## 3. 仮想サーバーとコンテナ基盤 - EC2、ECS+EC2、ECS+Fargate の役割と違い

### 先に押さえるポイント
- EC2単独はOS環境管理がメインで、コンテナはオプション的に導入可  
- ECS+EC2はコンテナ管理がECSに委任され、EC2は実行基盤  
- ECS+Fargateは完全サーバーレスのコンテナ実行に特化  
👉 **「何を管理するか」で分類すると理解しやすい**

### 3-1. 柔軟な仮想サーバー基盤 - EC2( Elastic Compute Cloud )
**EC2**はAWSの仮想サーバーで、OSイメージ( AMI )を選択して起動し、  
ユーザーがログインして自由に環境をカスタマイズできます。  
ソフトウェアのインストールやコード実行など、自由に環境を構築できます。

- **特徴**：OSレベルからシステム全体を管理・カスタマイズ可能  
- **主な利用場面**：パッケージインストールや独自のソフトウェア稼働など多用途  
- **利用できるコンテナイメージ**
   - OSイメージはAMI( Amazon Machine Image )から選択  
     ( 例：Amazon Linux、Ubuntu、Windows Server など )
     - 内部でDockerを用い、更にコンテナイメージを実行することも可能
     - プログラムやサービスを直接動かすことも一般的  

```plaintext
インターネット
   │
 VPC( プライベートネットワーク )
   └── EC2( 仮想環境 全手動管理 / OS選択+柔軟な環境カスタマイズ )
```

---

### 3-2. コンテナオーケストレーション - ECS + EC2クラスタ運用
**ECS( Elastic Container Service )** は、コンテナのデプロイ、管理、スケーリングを自動化するAWSのサービスで、EC2クラスタを基盤に複数コンテナを効率的に運用します。

- **特徴**：ECSがコンテナのスケーリング・フェイルオーバーを管理します。  
  - アクセス増加時はコンテナ数を自動的に増やし、負荷低下時は減らします。
  - 基盤となるクラスタ(EC2)のインフラ管理はユーザーが担当します。
- **主な利用場面**：マイクロサービスの構築・運用、大規模コンテナアプリケーション
- **利用できるコンテナイメージ**
   - Amazon ECR や Docker Hub などからのコンテナイメージ利用  
     自作イメージもアップロードして使用可能

```plaintext
インターネット
   │
 VPC( プライベートネットワーク )
   └── ECS + EC2( 仮想環境 クラスタ管理 / コンテナ運用 )
```

---

### 3-3. サーバーレスコンテナ実行 - ECS + Fargate
**Fargate**はECSのサーバーレス実行オプションで、サーバーやインフラの管理をAWSが代行します。  
ユーザーはコンテナイメージを指定し、アプリケーションに集中できます。

- **特徴**：サーバー管理不要の完全マネージド型コンテナ実行環境  
  - アクセス増加時はコンテナ数を自動的に増やし、負荷低下時は減らします。
  - 基盤となるクラスタのインフラ管理もAWSが自動対応します。
- **主な利用場面**：運用負荷軽減したいマイクロサービス、スケールが頻繁なアプリケーション  
- **利用できるコンテナイメージ**
   - Amazon ECR や Docker Hub などからコンテナイメージを利用  
     自作イメージもアップロードして使用可能

```plaintext
インターネット
   │
 VPC( プライベートネットワーク )
   └── ECS + Fargate( 仮想環境 自動クラスタ管理 / コンテナ運用 )
```

## 4. 選び方の本質
各サービスの違いは、「どこまでをユーザーが管理し、どこからをAWSに任せるか」にあります。  
👉 **この責任範囲の違いが、そのまま運用負荷と設計の自由度に直結します**

### 4-1. 比較一覧
用途や運用方法が異なります。特徴を理解して適切に選択しましょう。

| 項目 | EC2 | ECS+EC2 | ECS+Fargate |
|---|---|---|---|
| 利用可能な<br>コンテナイメージ | 任意のOSイメージを利用( AMI )<br>ログインしてのカスタマイズ( 自由度は高い ) | ECR、Docker Hubなどの外部リポジトリも利用可能<br>自作イメージも使用可能( アップロード ) | ( 左記に同じ ) |
| 管理負荷 | 高 | 中 | 低 |
| スケーリング・フェイルオーバー | - | ある程度 自動化 ※1 | 完全自動化 |
| 運用コスト | 中: EC2 | 中〜高: EC2台数次第 | 中: 使用分のみ |
| メリット | 柔軟に構築可能<br>構成が単純( クラスタ運用なし ) | コンテナ運用を自動化しつつ<br>柔軟に構成可能 ※2 | 運用負荷が低い |
| デメリット | OSの運用・保守が必要 | 基盤運用の管理が必要 | OS内部の細かな制御は難しい ※3 |

※1 **ある程度 自動化**
- EC2の空きリソース不足時は、EC2台数の調整も必要です。

※2 **コンテナ運用を自動化しつつ柔軟に構成可能**  
- ECSが配置・復旧・スケールを自動化
- EC2の台数や性能を調整しやすい
- 常時稼働ではコストを抑えやすい場合がある
- 既存のEC2運用ノウハウを活かしやすい

※3 **OS内部の細かな制御は難しい**  
- サーバーへ直接ログインして調査・設定はできない
- OS単位の細かなカスタマイズには不向き
- 常時稼働ではEC2より高コストになる場合がある

---

### 4-2. 管理負荷と責任範囲の違い

| 管理項目 | EC2 | ECS+EC2 | ECS+Fargate | サービス |
|---|:---:|:---:|:---:|---|
| ベース管理方法 | OS | コンテナイメージ | コンテナイメージ | EC2, ECR, ECS |
| ネットワーク・セキュリティ設定 | User | User | User | VPC, Security Group, ELB |
| インフラ管理<br>　サーバ切替<br>　パッチ適用 | User | User | AWS | EC2 |
| 基盤運用( 稼働維持 )<br>　スケーリング<br>　フェイルオーバー | - | User | AWS | ECS |
| ログ監視 | User | User | User ※1 | EC2, ECS, CloudWatch |

※1 Fargateではインフラ監視は不要ですが、アプリケーションログの収集・監視はユーザーがCloudWatchで確認する必要があります。

---

### 4-3. 結論：ユースケース別の選び方
用途に応じて最適なサービスは異なります。代表的な選択パターンを以下に示します。

| ケース | 推奨構成 | 理由 |
|---|---|---|
| とにかく自由に環境を作りたい | EC2 | OSレベルからカスタマイズ可能 |
| 既存のVMベース環境から移行したい | ECS + EC2 | 段階的にコンテナ化できる |
| マイクロサービスを効率的に運用したい | ECS + EC2 | スケーリングと管理のバランスが良い |
| サーバー管理をしたくない | ECS + Fargate | サーバーレスで運用負荷が最小 |
| スケールが頻繁・短期バッチ処理 | ECS + Fargate | 必要な分だけ起動できる |

## 5. 安全な通信の番人 - Security Group( セキュリティグループ )
Security GroupはVPC内リソースへのアクセスを制御する仮想ファイアウォールです。

- **役割**：許可する通信のみをホワイトリスト方式で設定  
- **設定例**：  
  - WebサーバーはインターネットからのHTTP( 80番ポート )を許可  
  - データベースは社内ネットワークのみアクセスを許可  
- **ポイント**：ブラックリスト方式ではなく、必要な通信だけを許可し、それ以外は遮断します  
- **注意**：過度に緩い設定はリスクとなるため、最小限の許可を徹底します

## 6. まとめ - ネットワーク・サーバー基盤の理解と実務活用
AWSのインフラの土台となる **VPC, EC2, Fargate, Security Group** を理解すると、  
システム全体の構造と安全な運用のイメージがつかめます。  

```plaintext
インターネット
    │
  VPC( プライベートネットワーク )
    └── Security Group( 通信制御 )
            ├── EC2          ( 仮想環境 全手動管理 / OS選択+柔軟な環境カスタマイズ )
            ├── ECS + EC2    ( 仮想環境 クラスタ管理 / コンテナ運用 )
            └── ECS + Fargate( 仮想環境 自動クラスタ管理 / コンテナ運用 )
```

- **ネットワーク( VPC )** で安全な空間を作る  
- **EC2** で仮想サーバーを自由に管理  
- **Fargate** で手軽にコンテナ運用  
- **Security Group** で通信を制御しリスクを低減  

この基本の組み合わせが、現代のクラウドシステムの強い味方になります。

## 7. 重要用語説明

| キーワード | 説明 |
|---|---|
| VPC<br>Virtual Private Cloud | AWSの仮想ネットワーク。ユーザー専用のネットワーク空間を作り安全に運用可能。 |
| Subnet( サブネット ) | VPC内を用途ごとに分割したネットワーク区画。Public / Private などに分けて利用する。 |
| EC2<br>Elastic Compute Cloud | 仮想サーバー。用途に応じたスペック選択が可能、多様な運用ができる。 |
| AMI<br>Amazon Machine Image | AWSでEC2インスタンスを起動するためのOSイメージ。 |
| ECS<br>Elastic Container Service | AWSのコンテナ管理サービス。 |
| Fargate | サーバー管理不要のコンテナ実行環境。インフラ設定なしでコンテナを運用可能。 |
| ECR<br>Elastic Container Registry | AWSのコンテナイメージリポジトリサービス。 |
| Docker Hub | Docker公式のパブリックコンテナレジストリ。多様なイメージが公開・共有されている。 |
| Failover( フェイルオーバー ) | 障害発生時に待機系へ自動切替し、サービス継続する仕組み。 |
| Scaling( スケーリング ) | 負荷に応じてサーバー数や性能を増減する仕組み。 |
| Security Group( セキュリティグループ ) | VPC内でのアクセス制御を行う仮想ファイアウォール。通信ルールを柔軟に設定できる。 |
| CI/CD( 継続的インテグレーション )<br>Continuous Integration/Continuous Delivery | 自動ビルド・テスト・デプロイの手法。 |
