# サーバーレスの基礎<br>( 管理不要で拡張自在なクラウド実行環境 )

サーバー管理の手間を省き、自動で拡張できる仕組みを学びます。  
AWSのサーバーレスサービスは、システムの運用負荷を減らしつつ迅速な開発を可能にします。

---

## 1. Lambda( ラムダ ) - 必要なときだけ動くサーバーレス関数

- **概要**  
  プログラムコードをサーバー管理なしで実行できるAWSのサービスです。  
  必要な処理だけをイベントに応じてすぐに実行でき、使った分だけ課金。

- **利用シーン例**  
  - Web APIのバックエンド処理  ( API Gatewayと連携 )
  - ファイルアップロード時の画像変換  ( S3と連携 )
  - 定期的なバッチ処理の代替  ( EventBridgeのスケジュール機能と連携 )

- **特徴**  
  - サーバー運用・保守不要  
  - イベント駆動型( 例: HTTPリクエスト、S3アップロード、タイマー実行 )  
  - 数秒～最大15分の処理に最適  
  - 料金は実行時間と処理回数に応じて発生( 無駄が少ない )  

---

## 2. API Gateway( APIゲートウェイ ) - APIを手軽に作成・管理

- **概要**  
  Web APIの入り口を作るAWSサービスです。  
  外部からのリクエストをLambdaなどのサービスへ安全かつ効率的に繋ぎます。

- **利用シーン例**  
  - Lambdaを利用したサーバーレスAPI構築  
  - モバイルアプリ・Webアプリと連携するAPIの公開  
  - 認証やアクセス制御の簡単実装  

- **特徴**  
  - APIの作成・公開・管理を簡単に  
  -認証・アクセス制御機能付き( IAM、Cognito連携可 )  
  - リクエストのルーティングや変換が可能  
  - 自動でスケールするため負荷増加に強い  

---

## 3. CloudWatch( クラウドウォッチ ) - 監視とログでシステムの健康管理

- **概要**  
  システムの「時間」や「状態」をきっかけに処理を動かし、その状況を監視する仕組みです。
  定期的な実行（バッチ処理）のスケジュール管理や、ログの収集を行います。

- **利用シーン例**
  - 定期的なバッチ処理の実行 ( 例: 毎日深夜0時にデータを集計 )  
  - Lambdaのエラーや実行時間の監視  
  - アラーム設定による自動通知( メール、SNSなど )  

- **特徴**  
  - スケジュール機能 「○分おき」「毎日○時」といった柔軟な実行指定が可能
  - ログの収集・保存・分析が一元管理  
  - メトリクスをリアルタイムに監視可能  
  - アラームを設定して問題を即時通知  
  - AWSサービス間の連携がスムーズ  

---

## 4. シンプルなサーバーレスの仕組み 図解

```plaintext
【パターンA:Web API】
        │
   API Gateway (API管理)
        │
     Lambda (処理実行)
        │
   CloudWatch (監視・ログ)
```
- ユーザーはAPI Gatewayを通じてLambdaにリクエスト  
- Lambdaが処理を行い、結果を返す  
- CloudWatchが動作状況を監視しログを保存  


```plaintext
  【パターンB】
        │
    EventBridge (タイマー)
        │
    Lambda (処理実行)
        │
    ClowdWatch (監視・ログ)    
```
- EventBridge（タイマー）が設定された時間になると自動で起動
- Lambdaが指定された処理（データの集計や掃除など）を実行
- CloudWatchが実行結果やエラーの有無を記録

---

## 5. 実務での役立ちポイント

- **運用負荷軽減**: サーバ管理が不要なため、運用コストと人手を大幅に削減  
- **スケーラビリティ**: 需要に応じ自動で拡張し、急なアクセス増加に対応可能  
- **コスト効率**: 利用量に応じた課金なので小規模から始めやすい  
- **迅速なサービス開発**: インフラ構築を気にせずコードに集中しやすい  

---

## 6. 注意点と改善案

- **Lambdaの最大実行時間**に注意( 最大15分 )  
  長時間処理が必要な場合は設計の見直しを検討  
- **API Gatewayのスロットリング( リクエスト制限 )**を理解し適切な設定を  
- **CloudWatchのログ保管コスト**は無視できないため、不要ログは適切に削除する  
- **セキュリティ面**  
  - APIの認証設定は必須  
  - Lambdaの実行権限は最小権限の原則で設定  

---

## 7. 重要用語説明

| キーワード               | 説明                                             |
|--------------------------|--------------------------------------------------|
| Lambda( ラムダ )          | AWSのサーバーレス関数実行サービス                |
| API Gateway( APIゲートウェイ ) | APIの作成・管理・公開を行うAWSサービス           |
| CloudWatch( クラウドウォッチ ) | AWSリソースの監視・ログ収集・アラーム通知サービス  |
| Serverless( サーバレス)        | サーバー運用不要でコードのみを実行する開発形態     |
| Scalability( スケーラビリティ )| システム負荷に応じて性能を自動拡張する能力         |
| CI/CD（継続的インテグレーション）<br>Continuous Integration/Continuous Delivery| プログラムのテストや本番反映を自動化する仕組み         |

---

## 8. ステップアップ：実践的な運用に向けて

本章ではサーバーレスの基本的な仕組みを学びましたが、実務ではこれらを「手作業」ではなく、ミスなく迅速に管理する手法が求められます。
- 動化（CI/CD）の導入
  プログラムを更新するたびに自動でテストを行い、Lambdaへ即座に反映させる仕組みを構築することで、リリース作業の安全性を高めます。
- AIによる開発サポート
  Amazon QなどのAIツールを活用し、インフラ構成の最適化やトラブルシューティングを効率化する手法も重要です。

> ※ **CI/CDの具体的な構築手順や、Amazon Qを活用した実践的な運用方法については、第4章「インフラ実践資料」にて詳しく解説します。**
