# IaC運用設計<br>( Tag設計、IAM設計、環境分離、変更管理、Single Source of Truth )
この章は実務寄りの内容です。  
IaCを「作る」だけでなく、「安全に運用するためのルール」を扱います。  

最初は全体像をつかめれば十分です。
👉 実務では「書けること」より「事故なく運用できること」が重要です。

## 1. Tag設計( 管理・コスト識別 )
Tagの基本については前章を参照してください。

IaC運用では、Tagは「付けるかどうか」ではなく、
**ルールとして統一すること**が重要です。

### 設計例( Parameter連動 )
```yaml
Tags:
  - Key: Env
    Value: !Ref Env
```

### 運用ルール
- 全リソースで共通Keyを使用する
- Env / Owner は必須とする
- コスト集計できるようにする
- 命名規則と整合性を持たせる

### よくある失敗
- リソースごとにKeyがバラバラ
- Owner未設定で管理不能
- Env未設定で環境識別不能

👉 Tagは「情報」ではなく「運用ルール」です。

## 2. 命名規則( Resource Naming )
リソース名は誰が見ても分かる形に統一します。

例：
- sales-dev-api-lambda
- sales-prod-log-bucket
- corp-stg-batch-role

推奨要素：
[システム]-[環境]-[用途]-[種別]

👉 名前だけで用途が分かる状態が理想です。

### よくある失敗
- 環境名が含まれていない
- 略語がバラバラで意味が分からない
- システムごとに命名ルールが違う

## 3. IAM設計( 最小権限 )
IAM は AWS 操作権限を制御します。  
IaC 運用では **誰が何を変更できるか** を明確にすることが重要です。

### 基本原則
- 開発者：dev環境のみ変更可
- 運用担当：本番閲覧 + 一部変更
- 管理者：限定人数のみ
- CI/CD：必要最低限の自動権限

### NG例
```plaintext
AdministratorAccess を全員付与
```

### 推奨
```plaintext
必要操作だけ許可
```

👉 事故の多くは権限過剰から発生します。

### よくある失敗
- とりあえず AdministratorAccess を付与
- 本番操作権限を全員に付与
- CI/CDに過剰権限を付与

## 4. 環境分離設計( dev / stg / prod )
開発環境と本番環境を分離しないと、テスト中に本番障害を起こす可能性があります。

### 一般例
| 環境 | 用途 |
|---|---|
| dev | 開発・試験 |
| stg | 本番前確認 |
| prod | 本番運用 |

### 分離方法
- AWSアカウント分離
- VPC分離
- Stack名分離
- Parameter分離
- Tag分離

```plaintext
dev-stack
stg-stack
prod-stack
```

👉 小規模でも最低限 dev / prod 分離推奨です。

### よくある失敗
- devとprodを同一環境で運用
- テスト中に本番リソースを変更
- 環境識別が名前やTagに含まれていない

## 5. 手作業変更禁止( IaC優先 )
AWS Console で直接変更すると、コードと実環境がズレます。

これを **ドリフト( drift )** と呼びます。

### 例
```plaintext
テンプレート上:
EC2 t3.micro

実環境:
手作業で t3.large
```

### 問題
- 再デプロイ時に戻る
- 原因不明障害になる
- レビュー履歴が残らない

### 原則
```plaintext
変更はテンプレート修正 → デプロイ
```

👉 緊急対応後も必ずコードへ戻すこと。

## 6. Single Source of Truth
唯一の正しい情報源をコードに統一する考え方です。

IaCでは、

```plaintext
AWS環境そのもの
ではなく

Git上のテンプレート
```

を正とします。

### 効果
- 履歴管理できる
- レビューできる
- 再現できる
- 属人化しにくい

### よくある失敗
- 実環境を正として扱ってしまう
- 設計書とコードが乖離する

## 7. 変更管理フロー

### 7-1 基本的な変更管理フロー
IaC の変更は、内容確認と承認を経て安全に反映することが重要です。  
実務では以下のような流れが基本になります。
```plaintext
テンプレート修正
↓
レビュー
↓
反映
```

#### ポイント
- いきなり本番環境へ変更しない
- 第三者レビューや承認フローを持つ
- 問題発生時のロールバック手順を準備する
- 変更理由を記録し、後から追えるようにする

👉 **変更を急ぐより、安全に反映できる流れを持つことが重要です。**

### 7-2 コード管理との連携
IaC テンプレートは、主に Git などのバージョン管理ツールと連携して管理します。  
変更履歴の保存、レビュー、差分確認がしやすくなり、チーム開発に向いています。  

詳細な Git の使い方は今後の章で説明しますので、  
ここでは **IaC はコードとして管理される** ことを理解できれば十分です。  
```plaintext
テンプレート修正
 ↓
Pull Request
 ↓
レビュー
 ↓
dev反映
 ↓
stg確認
 ↓
prod反映
```

#### ポイント
- 誰が何を変更したか履歴が残る
- 複数人でレビューしやすい
- 環境ごとに段階的に反映できる
- 手作業変更を減らしやすい

👉 **IaC と Git 管理は、実務で非常に相性の良い組み合わせです。**

### よくある失敗
- レビューなしで本番反映
- dev確認を飛ばしてprod反映
- 変更理由が残っていない

## 8. よくある失敗例
| 失敗 | 原因 |
|---|---|
| 環境分離なし | dev/stag/prod混在 |
| Tag未統一 | 管理不能 |
| IAM過剰権限 | 誰でも変更可能 |
| 手作業変更 | drift発生 |
| Git未管理 | 履歴追跡不可 |

## 9. よくある事故( 実務編 )
- 本番環境へ誤反映( dev/stag/prod混在 )
- 手作業変更でdrift発生
- IAM過剰権限による誤操作
- 不要リソース削除漏れによる課金継続
- 命名ルール不統一による運用混乱

## 10. まとめ
- Tag設計で管理しやすくする
- IAM設計で事故を防ぐ
- dev / stg / prod を分離する
- 手作業変更は禁止
- Git上のコードを正とする
- レビュー付き変更フローを持つ

👉 **IaC成功の本質は、テンプレート記述力より運用設計力です。**

## 11. 重要用語説明
| 用語 | 説明 |
|---|---|
| Drift( 構成差異 )<br>Configuration Drift | コード定義と実環境設定がズレること。 |
| Single Source of Truth( 正式情報一元化 ) | 正式な情報源を1か所に統一する考え方。 |
