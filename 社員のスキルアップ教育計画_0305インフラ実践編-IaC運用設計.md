# IaC運用設計<br>( Tag設計、IAM設計、環境分離、変更管理、Single Source of Truth )
この章では、IaCの実務運用ルールを扱います。  

最初は全体像をつかめれば十分です。  
👉 実務では「書けること」より「事故なく運用できること」が重要です。

本章では、以下の環境区分で運用することを前提とします。  
- dev：開発・試験
- stg：本番前確認
- prod：本番運用

環境ごとに変更範囲を分離することで、事故を防ぎます。

## 1. Tag設計( 管理・コスト識別 )
Tagの基本については前章を参照してください。

IaC運用では、Tagは「付けるかどうか」ではなく、
**ルールとして統一すること**が重要です。

[ 設計例 ]
```yaml
Tags:
  - Key: Env
    Value: !Ref Env
```

[ 運用ルール ]
- 全リソースで共通Keyを使用する
- Env / Owner は必須とする
- コスト集計できるようにする
- 命名規則と整合性を持たせる

[ よくある失敗 ]
- リソースごとにKeyがバラバラ
- Owner未設定で管理不能
- Env未設定で環境識別不能

👉 Tagは「情報」ではなく「運用ルール」です。

## 2. 命名規則( Resource Naming )
リソース名は誰が見ても分かる形に統一します。

例：
- sales-dev-api-lambda
- sales-prod-log-bucket
- corp-stg-batch-role

[ 命名ルール ]
```plaintext
[システム]-[環境]-[用途]-[種別]
```

[ 設計ルール ]
- 名前には「変わりにくい情報」を持たせる
- Ownerなどの管理情報は、名前ではなくTagで管理する

👉 名前だけで用途が分かる状態が理想です。

[ よくある失敗 ]
- 環境名が含まれていない
- 略語がバラバラで意味が分からない
- システムごとに命名ルールが違う

## 3. IAM設計( 最小権限 )
IAM は AWS 操作権限を制御します。  
IaC 運用では **誰が何を変更できるか** を明確にすることが重要です。

[ 基本原則 ]
- 必要操作だけ許可
  - 管理者：限定人数のみ
  - 開発者：dev環境のみ変更可
  - CI/CD：必要最低限の自動権限
  - 運用担当：prod環境は参照のみ

[ 設計例 ]
```plaintext
管理者：
  dev/stg/prod全環境の全操作が可能( Administrator )
  ※通常は使用せず、障害対応など例外時のみ利用

開発者：
  dev/stg環境のみ作成・更新可能( prodは通常不可 )

CI/CD：
  dev/stg/prod全環境のデプロイ対象スタックの更新のみ許可
  ※prod環境の反映方法は運用方針により決定する

運用担当：
  prod環境を参照のみ
```

👉 「誰がどこまで触れるか」を明確に分けるのが基本です。

[ NG例 ]
```plaintext
AdministratorAccess を全員付与
```

👉 事故の多くは権限過剰から発生します。

[ よくある失敗 ]
- とりあえず AdministratorAccess を付与
- 本番操作権限を全員に付与
- CI/CDに過剰権限を付与

## 4. Single Source of Truth( IaCの原則 )
IaCでは、実環境ではなく「コード( テンプレート )」を正とします。
唯一の正しい情報源をコードに統一する考え方です。

[ 効果 ]
- 履歴管理できる
- レビューできる
- 再現できる
- 属人化しにくい

AWS Console で直接変更すると、コードと実環境がズレます。 
以下はその例です。
[ 手作業 例 ]
```plaintext
テンプレート上:
  EC2 t3.micro

実環境:
  手作業で t3.large
```
これを **ドリフト( drift )** と呼びます。

[ 問題 ]
- コードと実環境がズレる( drift )
- 再デプロイ時に意図しない変更が発生
- 原因不明障害になる

[ ルール ]
```plaintext
変更はテンプレート修正 → デプロイ
```

👉 緊急対応後も必ずコードへ戻すこと。

[ よくある失敗 ]
- 実環境を正として扱ってしまう
- 設計書とコードが乖離する

## 5. 変更管理フロー
IaCの変更は、レビューと履歴管理を前提として安全に反映します。  
実務では、Gitと連携した変更フローで運用するのが一般的です。
```plaintext
テンプレート修正
 ↓
Pull Request
 ↓
レビュー
 ↓
dev反映
 ↓
stg確認
 ↓
prod反映
```

[ ポイント ]
- いきなり本番環境へ変更しない
- 第三者レビューや承認フローを持つ
- 問題発生時のロールバック手順を準備する
- 変更理由を記録し、後から追えるようにする

👉 **変更を急ぐより、安全に反映できる流れを持つことが重要です。**

[ 効果 ]
- 誰が何を変更したか履歴が残る
- 複数人でレビューしやすい
- 環境ごとに段階的に反映できる
- 手作業変更を減らしやすい

[ よくある失敗 ]
- レビューなしで本番反映
- dev確認を飛ばしてprod反映
- 変更理由が残っていない

## 6. よくある失敗と事故
| 失敗(原因) | 起こる問題(結果) |
|---|---|
| 変更範囲の分離なし | 本番誤更新 / dev・stg・prod混在 |
| Tag未統一 | 管理不能 / コスト不明 |
| IAM過剰権限 | 誰でも変更可能 / 誤操作 |
| 手作業変更 | drift発生 / 不整合 |
| Git未管理 | 履歴追跡不可 / 引継ぎ不能 |
| 命名ルール不統一 | 運用混乱 / 調査困難 |

## 7. まとめ
[ リソース管理 ]  
- 環境(dev/stg/prod)ごとに変更範囲を分離する
- Tag付けをルールとして統一する
- 命名ルールを統一する

[ 権限管理 ]  
- IAMは最小権限で設計する

[ 変更管理 ]  
- Single Source of Truth(コードを正)を守る
- 手作業変更は禁止し、変更はレビュー付きで反映する

👉 IaC成功の本質は「運用設計力」です。

## 8. 重要用語説明
| 用語 | 説明 |
|---|---|
| Drift( 構成差異 )<br>Configuration Drift | コード定義と実環境設定がズレること。 |
| Single Source of Truth( 正式情報一元化 ) | 正式な情報源を1か所に統一する考え方。 |
