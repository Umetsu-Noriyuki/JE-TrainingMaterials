# 全エンジニア共通知識_S0102 Git-復旧編

## 1. この資料の目的
この資料は、Git / GitHub 操作でミスやトラブルが起きた時に、  
作業内容を壊さず、安全に復旧するための基礎を学ぶ資料です。

Git は変更履歴を管理する仕組みですが、操作を誤ると、  
「どの変更が残っているのか」「どこまで戻してよいのか」が分かりにくくなることがあります。

そのため、この資料では、いきなり危険な操作を覚えるのではなく、  
まず状況を確認し、必要な変更を守りながら復旧する考え方を扱います。

## 2. この資料で扱うこと
この資料では、次のような「困った時」の対応を扱います。

- pull したら conflict が発生した
- 作業する branch を間違えた
- commit する内容を間違えた
- commit message を修正したい
- push 前の commit を直したい
- push 後の変更を取り消したい
- reset / revert の違いを知りたい
- 過去の状態を reflog で探したい
- 必要な commit だけ cherry-pick したい

## 3. 復旧操作で大事な考え方
Git の復旧操作では、最初に次の3点を確認します。

| 確認すること | 説明 |
|---|---|
| 変更は保存されているか | 未commitの作業が消えないように確認する |
| push 済みか | Remote環境( GitHub )に共有済みか確認する |
| 他の人に影響するか | 共有済みの履歴を書き換えないように注意する |

特に、push 済みの履歴を変更する操作は、他の人の作業に影響する可能性があります。  
そのため、共有済みの変更を取り消す場合は、基本的に `revert` を使います。

一方、push 前のローカル作業であれば、`amend` や `reset` で整理できる場合があります。

## 4. 復旧前に確認する基本コマンド
復旧操作を行う前に、まず現在の状態を確認します。

```bash
git status
````

現在の branch、変更中のファイル、commit 前の変更があるかを確認します。

```bash
git branch
```

今どの branch にいるかを確認します。

```bash
git log --oneline --graph --decorate --all
```

commit の流れと branch の位置を確認します。

```bash
git remote -v
```

Remote環境( GitHub )の接続先を確認します。

## 5. 復旧操作の基本方針
困った時は、次の順番で考えます。

```plaintext
1. まず現在の状態を確認する
   ↓
2. 未commitの変更があるか確認する
   ↓
3. push 済みか確認する
   ↓
4. 他の人に影響する操作か確認する
   ↓
5. 安全な復旧方法を選ぶ
```

復旧操作は、焦って実行すると状況が悪化することがあります。
まず `git status` で状態を確認し、必要であれば作業内容を退避してから対応します。

以下を、現在の末尾にそのまま追加していく形で良いと思います。
S0101の流れを踏襲して、「まず現場で遭遇しやすい順」に並べています。 

## 6. fetch / pull の違い
GitHub 上の最新変更を取得する際に、`fetch` と `pull` を使用します。

### 6-1. fetch
Remote環境( GitHub )の最新情報だけ取得します。  
現在の作業内容は変更されません。

```bash
$ git fetch
````

[ イメージ ]
```plaintext
GitHub の最新情報を取得
↓
まだ merge はしない
↓
安全に差分確認できる
```

### 6-2. pull
Remote環境( GitHub )の最新情報を取得し、現在の branch に反映します。
```bash
git pull
```

[ イメージ ]
```plaintext
GitHub の最新情報を取得
↓
自動 merge
↓
conflict が発生する場合がある
```

### 6-3. まず fetch が安全
作業中にいきなり `pull` すると、  
conflict や意図しない merge が発生することがあります。

そのため、まず `fetch` で差分確認する方が安全です。
```bash
git fetch
git log --oneline --graph --decorate --all
```

## 7. conflict対応

### 7-1. conflictとは
同じ箇所を別々に変更した場合、
Git が自動 merge できず、conflict が発生します。

```plaintext
CONFLICT (content): Merge conflict in sample.txt
```

### 7-2. よくある発生タイミング
* pull 時
* merge 時
* rebase 時
* 同じファイルを複数人が編集した時

### 7-3. 基本対応
```plaintext
1. conflict発生
   ↓
2. conflictファイル確認
   ↓
3. 内容を手修正
   ↓
4. add
   ↓
5. commit
```

### 7-4. conflict記号
```plaintext
<<<<<<< HEAD
現在の変更
=======
取得した変更
>>>>>>> feature/sample
```

### 7-5. 解消後
修正後に add / commit を行います。
```bash
git add .
git commit
```

### 7-6. VSCodeでの対応
VSCode では conflict 箇所に以下が表示されます。
* Current Change
* Incoming Change
* Accept Both Changes

GUIで確認しながら安全に修正できます。

## 8. branch を間違えた時

### 8-1. commit 前の場合
変更だけ別branchへ移動したい場合です。
```bash
git switch 正しいbranch
```

変更内容は保持されたまま移動できます。

### 8-2. commit 後の場合
誤ったbranchで commit した場合です。
```plaintext
誤branchでcommit
↓
正branchへcommit移動
↓
誤branch側を戻す
```

### 8-3. cherry-pick で移動
必要な commit だけ別branchへ適用できます。
```bash
git cherry-pick {commit id}
```

### 8-4. 元branchを戻す
push 前なら reset で戻せます。
```bash
git reset --hard HEAD~1
```

※ push 済みの場合は、基本的に `revert` を使用します。

## 9. commit を間違えた時

### 9-1. commit message 修正
直前の commit message を修正します。
```bash
git commit --amend -m "修正後message"
```

### 9-2. add 漏れ修正
直前commitへ追加できます。
```bash
git add .
git commit --amend
```

### 9-3. amend の注意点
`amend` は commit を作り直します。

そのため、push 済みの commit に対して使用すると、  
履歴不整合が発生する場合があります。

基本的に、push 前のみ使用します。

## 10. push を間違えた時

### 10-1. push 前なら reset
ローカルだけなら reset で戻せます。
```bash
git reset --soft HEAD~1
```

### 10-2. push 後なら revert
共有済み履歴は、基本的に `revert` を使用します。
```bash
git revert {commit id}
```

### 10-3. revertとは
対象commitを打ち消す commit を追加します。
```plaintext
過去commitを消す
ではなく
↓
取り消しcommitを追加する
```

履歴を壊さないため、安全です。

## 11. reset の違い

### 11-1. soft
commitだけ戻します。
```bash
git reset --soft HEAD~1
```

```plaintext
commit : 戻る
add    : 保持
変更   : 保持
```

### 11-2. mixed
add を解除します。
```bash
git reset HEAD~1
```

```plaintext
commit : 戻る
add    : 戻る
変更   : 保持
```

### 11-3. hard
変更自体を破棄します。
```bash
git reset --hard HEAD~1
```

```plaintext
commit : 戻る
add    : 戻る
変更   : 消える
```

### 11-4. hard の注意
未保存変更が消えるため注意が必要です。

特に、push 済み履歴への使用は避けます。

## 12. reflog

### 12-1. reflogとは
HEAD の移動履歴を確認できます。
```bash
git reflog
```

### 12-2. reset事故から戻す
```plaintext
間違えてreset
↓
reflogで過去位置確認
↓
元へ戻す
```

### 12-3. 復旧例
```bash
git reflog
git reset --hard HEAD@{1}
```

### 12-4. reflog は最後の保険
消えたように見える commit も、
reflog から復旧できる場合があります。

## 13. cherry-pick

### 13-1. cherry-pickとは
必要な commit だけ別branchへ適用します。

```bash
git cherry-pick {commit id}
```

### 13-2. よくある用途
* 別branchへ修正だけ反映
* hotfix を複数branchへ反映
* 誤branch commit の移動

### 13-3. conflict注意
cherry-pick 中も conflict が発生する場合があります。

その場合は通常 conflict と同様に解消します。

## 14. 復旧時によく使う確認コマンド

### 状態確認
```bash
git status
```

### branch確認
```bash
git branch
```

### commit履歴確認
```bash
git log --oneline --graph --decorate --all
```

### 最新操作履歴確認
```bash
git reflog
```

## 15. まとめ
Git の復旧操作では、  
まず「今どういう状態か」を確認することが重要です。

特に次の3点を意識します。
* push 済みか
* 他人へ影響するか
* 変更が消えないか

共有済み履歴は `revert`、  
ローカル整理は `amend` や `reset` を基本に考えると、安全に運用しやすくなります。

また、困った時は焦って操作せず、  
まず `git status` と `git log` を確認する習慣が重要です。
