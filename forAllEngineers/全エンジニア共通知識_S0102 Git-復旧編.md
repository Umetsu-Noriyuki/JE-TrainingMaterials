# 全エンジニア共通知識_S0102 Git-復旧編

## 1. この資料の目的
この資料は、Git / GitHub 操作でミスやトラブルが起きた時に、  
作業内容を壊さず、安全に復旧するための基礎を学ぶ資料です。

Git は変更履歴を管理する仕組みですが、操作を誤ると、  
「どの変更が残っているのか」「どこまで戻してよいのか」が分かりにくくなることがあります。

そのため、この資料では、いきなり危険な操作を覚えるのではなく、  
まず状況を確認し、必要な変更を守りながら復旧する考え方を扱います。

## 2. 復旧操作の基本方針
現在の状態を確認し、push前かどうかで対応を分けます。
```plaintext
 push前かどうか
     ├─ Yes ⇒ local復旧
     │            GitHub未共有の local変更や履歴を整理・復旧する操作
     └─ No  ⇒ GitHub共有後復旧
                   GitHub共有後の履歴を安全に復旧する操作
```

Git の復旧操作は、  
「まだ GitHub へ共有されていない local作業」  
と、  
「GitHub共有後の履歴」  
で、使用する考え方やコマンドが変わります。

そのため、この資料では、  
local復旧 と GitHub共有後復旧 の2つに分けて、  
以下の内容を中心に扱います。
| 種類 | 内容 | 使用コマンド | 章 |
|---|---|---|---|
| local作業中の復旧 | conflict対応 | fetch / pull / merge | 4章 |
| local作業中の復旧 | branch間違い | switch / cherry-pick | 5章 |
| local作業中の復旧 | commit修正 | amend | 6章 |
| local作業中の復旧 | push前の履歴整理 | reset | 7章 |
| GitHub共有後の復旧 | push後の変更取り消し | revert | 7章 |
| local作業中の復旧 | 過去履歴復旧 | reflog | 8章 |
| local作業中の復旧 | 一時退避 | stash | 9章 |
| local作業中の復旧 | commit移送 | cherry-pick | 10章 |

## 3. 確認コマンド
現在の状態を確認し、  
local復旧か GitHub共有後復旧かを判断するためのコマンドです。

```bash
# 現在の branch、変更中のファイル、commit 前の変更があるかを確認
$ git status
On branch feature/sample               # 現在作業中の branch

Changes not staged for commit:         # 未commit変更が存在
  modified: sample.txt                 #   ファイル名も判明

# 今どの branch にいるかを確認
$ git branch
- feature/sample                       # 現在作業中の branch
  main

# commit の流れと branch の位置を確認
$ git log --oneline --graph --decorate --all
- a1b2c3d (HEAD -> feature/sample)     # HEADが作業中branch
| * e4f5g6h (origin/main)              # GitHub側 main branch の最新位置
|/
- z9y8x7w                              # branch分岐元

# Remote環境( GitHub )の接続先を確認
$ git remote -v
origin  https://github.com/example/sample.git (fetch)    # origin が GitHub 接続先
origin  https://github.com/example/sample.git (push)     # origin が GitHub 接続先

# 最新操作履歴確認
$ git reflog
```

## 4. conflict対応

### 4-1. pull前の fetch確認
GitHub 上の最新変更を取得する際に、`fetch` と `pull` を使用します。

#### [ fetch ]
Remote環境( GitHub )の最新情報だけ取得します。  
現在の作業内容は変更されません。
```bash
# GitHub の最新情報を取得
#     ↓
#   まだ merge はしない ⇒ 安全に差分確認できる
$ git fetch
```

#### [ pull ]
Remote環境( GitHub )の最新情報を取得し、現在の branch に反映します。
```bash
# GitHub の最新情報を取得
#     ↓
#   自動 merge ⇒ conflict が発生する場合がある
$ git pull
```

#### [ まず fetch が安全 ]
GitHub側の変更を取り込む際は、  
conflict や意図しない merge が発生することがあります。

特に、  
- pull 時
- branch merge 時
- PR merge 時

などで発生することがあります。

そのため、まず `fetch` で差分確認する方が安全です。
```bash
$ git fetch
$ git log --oneline --graph --decorate --all
```

### 4-2. conflictとは
同じ箇所を別々に変更した場合、
Git が自動 merge できず、conflict が発生します。

#### [ 発生時の対応の流れ ]
```plaintext
1. conflict発生
   CONFLICT (content): Merge conflict in sample.txt
   ↓
2. conflict が発生したファイルを確認しながら修正 ⇒ 修正後は通常の反映方法と同じ( add + commit )
   <<<<<<< HEAD
   現在の変更
   =======
   取得した変更
   >>>>>>> feature/sample
```

### 4-3. 修正方法

#### [ 基本の修正方法 ]
conflict は「どの変更を残すか」を人間が判断して修正します。
```plaintext
<<<<<<< HEAD
現在の変更
=======
取得した変更
>>>>>>> feature/sample
```
不要な行を削除し、  
最終的に残したい内容へ修正します。

修正後は通常通り、
```bash
$ git add .
$ git commit
```

を実施します。

#### [ VSCodeでの対応 ]
対象ソースの conflict 箇所所運部に以下が表示されます
GUIで選択しながら安全に修正できます。
<div align="center"><img src="./img/S0101git_vscode-conflict.png" width="500"></div>

- 現在に変更を取り込む( Current Change )
- 入力側の変更を取り込む( Incoming Change )
- 両方の変更を取り込む( Accept Both Changes )
- 変更の比較( xxx )

#### [ GitHubでの対応 ]
GitHub の PR merge 時も、VSCodeと同じように修正出来る場合があります。
この場合、VSCodeと同じように GUIで選択しながら安全に修正できます。
そして、GitHub にすでに上がっているため、push操作がありません。

## 5. branch を間違えた時

### 5-1. commit 前の場合

変更だけ別branchへ移動したい場合です。

```bash
$ git switch 正しいbranch
```

変更内容は保持されたまま移動できます。

### 5-2. commit 後の場合

誤ったbranchで commit した場合です。

```plaintext
誤branchでcommit
↓
正branchへcommit移動
↓
誤branch側を戻す
```

### 5-3. cherry-pick で移動

必要な commit だけ別branchへ適用できます。

```bash
$ git cherry-pick {commit id}
```

### 5-4. 元branchを戻す

push 前なら reset で戻せます。

```bash
$ git reset --hard HEAD~1
```

※ push 済みの場合は、基本的に `revert` を使用します。

## 6. commit を間違えた時

### 6-1. commit message 修正

直前の commit message を修正します。

```bash
$ git commit --amend -m "修正後message"
```

### 6-2. add 漏れ修正

直前commitへ追加できます。

```bash
$ git add .
$ git commit --amend
```

### 6-3. amend の注意点

`amend` は commit を作り直します。

そのため、push 済みの commit に対して使用すると、
履歴不整合が発生する場合があります。

基本的に、push 前のみ使用します。

## 7. push を間違えた時

### 7-1. push 前なら reset

ローカルだけなら reset で戻せます。

```bash
$ git reset --soft HEAD~1
```

### 7-2. push 後なら revert

共有済み履歴は、基本的に `revert` を使用します。

```bash
$ git revert {commit id}
```

### 7-3. revertとは

対象commitを打ち消す commit を追加します。

```plaintext
過去commitを消す
ではなく
↓
取り消しcommitを追加する
```

履歴を壊さないため、安全です。

## 8. reset の違い

### 8-1. soft

commitだけ戻します。

```bash
$ git reset --soft HEAD~1
```

```plaintext
commit : 戻る
add    : 保持
変更   : 保持
```

### 8-2. mixed

add を解除します。

```bash
$ git reset HEAD~1
```

```plaintext
commit : 戻る
add    : 戻る
変更   : 保持
```

### 8-3. hard

変更自体を破棄します。

```bash
$ git reset --hard HEAD~1
```

```plaintext
commit : 戻る
add    : 戻る
変更   : 消える
```

### 8-4. hard の注意

未保存変更が消えるため注意が必要です。

特に、push 済み履歴への使用は避けます。

## 9. reflog

### 9-1. reflogとは

HEAD の移動履歴を確認できます。

```bash
$ git reflog
```

### 9-2. reset事故から戻す

```plaintext
間違えてreset
↓
reflogで過去位置確認
↓
元へ戻す
```

### 9-3. 復旧例

```bash
$ git reflog
$ git reset --hard HEAD@{1}
```

### 9-4. reflog は最後の保険

消えたように見える commit も、
reflog から復旧できる場合があります。

## 10. cherry-pick

### 10-1. cherry-pickとは

必要な commit だけ別branchへ適用します。

```bash
$ git cherry-pick {commit id}
```

### 10-2. よくある用途
- 別branchへ修正だけ反映
- hotfix を複数branchへ反映
- 誤branch commit の移動

### 10-3. conflict注意
cherry-pick 中も conflict が発生する場合があります。  
その場合は通常 conflict と同様に解消します。

## 11. まとめ
Git の復旧操作では、  
まず「今どういう状態か」を確認することが重要です。

特に、  
「push前の local作業なのか」  
「GitHub共有後なのか」  
を意識することが重要です。

共有済み履歴は `revert`、  
ローカル整理は `amend` や `reset` を基本に考えると、安全に運用しやすくなります。

また、困った時は焦って操作せず、  
まず `git status` と `git log` を確認する習慣が重要です。
