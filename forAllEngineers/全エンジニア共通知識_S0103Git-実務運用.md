# 全エンジニア共通知識_S0103 Git-実務運用編

## 1. この資料の目的

Git操作自体は覚えていても、
実務では「どう運用するか」で迷うことが多くあります。

例えば、

* branch をどう切るか
* commit をどの粒度で分けるか
* PR をどう出すか
* review をどう行うか
* main branch をどう守るか

などは、チームごとに運用ルールがあります。

この章では、
「Gitコマンドの使い方」ではなく、
チーム開発での実務運用の考え方を整理します。

```plaintext
S0101 ... Git/GitHub基礎操作
S0102 ... Gitトラブル復旧
S0103 ... Git実務運用
```

---

## 2. Git運用は「履歴管理」と「共同作業」

Git は単なるバックアップではありません。
チーム全体で、

* 誰が
* 何を
* なぜ変更したか

を共有・管理するための仕組みです。

そのため実務では、

```plaintext
動けばOK
```

ではなく、

```plaintext
他人が理解できる履歴になっているか
```

が重要になります。

---

## 3. branch運用

### 3-1. branchとは

branch は、変更作業を分離するための作業ラインです。

```plaintext
main
 ├ feature/login
 ├ feature/user-edit
 └ bugfix/header
```

複数人が同時に作業しても、
変更が混ざりにくくなります。

---

### 3-2. main へ直接commitしない

実務では通常、

```plaintext
main branch
```

へ直接 commit / push は行いません。

理由:

* 本番相当コードであることが多い
* 不具合混入リスクが高い
* review無し変更を防ぐため

一般的には、

```plaintext
作業branch
↓
PR作成
↓
review
↓
merge
```

の流れになります。

---

### 3-3. branch命名例

よくある命名例:

| branch名        | 用途     |
| -------------- | ------ |
| feature/login  | 新機能    |
| bugfix/header  | 不具合修正  |
| hotfix/auth    | 緊急修正   |
| release/v1.2.0 | リリース調整 |
| develop        | 開発統合用  |

---

## 4. commit粒度

### 4-1. commitは「意味単位」で分ける

commit は、

```plaintext
あとから履歴を読めること
```

が重要です。

悪い例:

```plaintext
いろいろ修正
```

良い例:

```plaintext
ログイン画面追加
ヘッダーデザイン修正
不要API削除
```

---

### 4-2. commitを細かくしすぎる問題

細かすぎる commit も実務では読みにくくなります。

例:

```plaintext
typo修正
空白修正
改行修正
```

基本は、

```plaintext
ひとつの意味を持つ変更
```

を1commitにまとめます。

---

### 4-3. commit途中保存だけをpushしない

実務では以下のような commit を避けることがあります。

```plaintext
wip
tmp
途中
```

理由:

* reviewしづらい
* 履歴が汚れる
* squash前提運用でない場合に残る

※ WIP(commit途中保存) を使うチームもあります

---

## 5. commit message

### 5-1. commit messageの役割

commit message は、

```plaintext
何を変更したか
```

を履歴として残す説明文です。

---

### 5-2. よくある形式

実務では Prefix を付けることがあります。

| Prefix   | 意味     |
| -------- | ------ |
| feat     | 機能追加   |
| fix      | 不具合修正  |
| refactor | リファクタ  |
| docs     | ドキュメント |
| test     | テスト    |
| chore    | 雑務系    |

例:

```plaintext
feat: ログイン画面追加
fix: ヘッダー崩れ修正
docs: README更新
```

---

### 5-3. 「なぜ」を残す

可能であれば、

```plaintext
なぜ変更したか
```

も重要です。

悪い例:

```plaintext
fix: 修正
```

良い例:

```plaintext
fix: 二重送信時に注文が重複する問題を修正
```

---

## 6. Pull Request( PR )運用

### 6-1. PRとは

PR( Pull Request ) は、

```plaintext
この変更を取り込みたいです
```

というレビュー依頼です。

GitHub 上で、

* 差分確認
* コメント
* review
* approve

などを行います。

---

### 6-2. PRは小さくする

巨大PRは review が困難になります。

悪い例:

```plaintext
50ファイル変更
3000行変更
```

良い例:

```plaintext
ログイン機能だけ
```

基本:

```plaintext
PRは小さいほど reviewしやすい
```

---

### 6-3. PR説明に書くこと

よくある記載:

* 変更概要
* 動作確認内容
* 影響範囲
* 確認してほしい点
* 関連Issue

例:

```plaintext
■ 概要
ログイン画面追加

■ 確認内容
正常ログイン確認済

■ 影響範囲
認証機能周辺
```

---

## 7. review文化

### 7-1. reviewは「人格否定」ではない

review は、

```plaintext
コード品質向上
```

が目的です。

```plaintext
人を責める
```

ものではありません。

---

### 7-2. 指摘されるのは普通

実務では、

```plaintext
レビュー指摘ゼロ
```

のほうが珍しいこともあります。

そのため、

```plaintext
指摘 = ダメ
```

ではありません。

---

### 7-3. review観点例

reviewでは以下を確認します。

* バグがないか
* 可読性
* 命名
* 保守性
* セキュリティ
* 性能
* 設計整合性

---

## 8. approve と merge

### 8-1. approveとは

approve は、

```plaintext
変更取り込みOK
```

という意味です。

チームによっては、

```plaintext
1人approve必須
2人approve必須
```

などのルールがあります。

---

### 8-2. merge前の確認

merge 前には、

* CI成功
* conflict無し
* review完了
* 動作確認

などを確認します。

---

## 9. main保護

### 9-1. main保護とは

GitHub では、

```plaintext
main branch
```

を保護できます。

例:

* 直接push禁止
* approve必須
* CI成功必須
* force push禁止

---

### 9-2. なぜ必要か

main は本番影響に近いことが多いため、
誤操作防止が重要です。

```plaintext
誤push
↓
main破壊
↓
全員影響
```

を防ぐ目的があります。

---

## 10. squash merge

### 10-1. squash mergeとは

複数 commit を、

```plaintext
1commit
```

にまとめて merge する機能です。

```plaintext
作業commit
 ├ fix
 ├ typo
 ├ debug
 └ cleanup

↓ squash

feat: ログイン機能追加
```

---

### 10-2. squash mergeの利点

履歴が整理されやすくなります。

特に、

```plaintext
途中commitが多いチーム
```

では有効です。

---

### 10-3. 注意点

細かい作業履歴は消えます。

そのため、

```plaintext
詳細履歴を残したいか
履歴を整理したいか
```

で運用が変わります。

---

## 11. release運用

### 11-1. releaseとは

release は、

```plaintext
利用者へ公開する単位
```

です。

例:

```plaintext
v1.0.0
v1.1.0
v2.0.0
```

---

### 11-2. release branch運用例

大規模開発では、

```plaintext
develop
↓
release branch
↓
最終確認
↓
main
```

のような運用もあります。

---

### 11-3. 緊急修正( hotfix )

本番障害時などは、

```plaintext
main
↓
hotfix branch
↓
修正
↓
即release
```

となることがあります。

---

## 12. tag運用

### 12-1. tagとは

tag は、

```plaintext
このcommitが特定バージョン
```

であることを示す印です。

例:

```plaintext
v1.0.0
v1.1.0
```

---

### 12-2. tagの用途

よくある用途:

* リリース地点管理
* rollback基準
* デプロイ対象指定
* 過去版再現

---

### 12-3. release と tag

実務では、

```plaintext
release時にtag付与
```

することが多いです。

```plaintext
release v1.2.0
↓
tag: v1.2.0
```

---

## 13. 実務で重要な考え方

Git運用で重要なのは、

```plaintext
自分だけ分かれば良い
```

ではなく、

```plaintext
チーム全員が理解できること
```

です。

そのため、

* branch運用
* commit整理
* PR文化
* review文化
* main保護

などを含めて、

```plaintext
安全に共同開発する仕組み
```

としてGitを利用します。

---

## 14. まとめ

* Git実務では「運用ルール」が重要
* branch を分けて安全に作業する
* commit は意味単位で整理する
* PR は小さく分かりやすくする
* review は品質向上のために行う
* main branch は保護する
* squash merge で履歴整理することがある
* release / tag で公開版を管理する
