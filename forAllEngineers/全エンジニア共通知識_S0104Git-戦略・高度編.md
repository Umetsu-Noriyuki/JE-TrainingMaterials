# 全エンジニア共通知識_S0104 Git-戦略・高度編

## 1. この資料の目的

この資料では、Git の操作方法ではなく、  
チームやプロジェクトに合わせて Git運用をどう設計するかを学びます。

S0101 / S0102 では、個人が Git/GitHub を操作し、  
困った時に復旧するための基本を扱いました。

この章では、現場や組織によって変わる Git運用方針を整理します。

```plaintext
S0101 ... Git/GitHub の基本操作
S0102 ... Gitトラブルからの復旧
S0103 ... 現場で迷わない実務運用
S0104 ... チーム・組織単位のGit運用設計
```

Git運用に正解は1つではありません。
重要なのは、プロジェクトの規模、リリース頻度、レビュー体制、CI/CDの有無に合わせて、
無理のない運用を選ぶことです。

この資料では、以下を扱います。

* ブランチ戦略
* Git Flow
* GitHub Flow
* trunk based development
* fork運用
* monorepo
* submodule
* CI/CD連携

## 2. Git運用設計とは

Git は単なるバージョン管理ツールではなく、  
チーム開発ルールを支える基盤でもあります。

同じ Git を使っていても、
- 少人数開発
- 大規模開発
- 頻繁リリース
- 厳格レビュー
- OSS開発

など、状況によって最適な運用は変わります。

例えば、
```plaintext
小規模チーム
→ シンプル重視

大規模チーム
→ 厳格なbranch管理重視

高頻度リリース
→ merge待ち削減重視
````

Git戦略では、以下を設計します。

* branch の分け方
* merge方法
* releaseタイミング
* reviewフロー
* CI/CD連携
* Repository構成

# 3. ブランチ戦略

branch戦略とは、
「どのbranchをどう使うか」を決める運用設計です。

branch戦略が曖昧だと、

* merge衝突増加
* release混乱
* review停滞
* hotfix事故

などが発生しやすくなります。

## 3-1. よくあるbranch構成

```plaintext
main
 ├ develop
 │   ├ feature/login
 │   ├ feature/cart
 │   └ feature/order
 └ hotfix/urgent-fix
```

## branch役割例

| branch    | 用途        |
| --------- | --------- |
| main      | 本番反映コード   |
| develop   | 開発統合      |
| feature/* | 個別開発      |
| release/* | release調整 |
| hotfix/*  | 緊急修正      |

## branch戦略で重要なこと

重要なのは、

* branch数を増やしすぎない
* merge責任を明確にする
* releaseフローを固定化する

ことです。

branchを細かくしすぎると、
管理コストが急増する場合があります。

# 4. Git Flow

Git Flow は、
比較的伝統的な大規模開発向けbranch戦略です。

## 基本構成

```plaintext
main
 └ develop
     ├ feature/*
     ├ release/*
     └ hotfix/*
```

## 特徴

* branch役割が明確
* release管理しやすい
* 大人数向け
* 長期開発向け

一方で、

* branch数が増える
* merge回数増加
* 運用が重くなりやすい

という特徴もあります。

## 向いているケース

* エンタープライズ開発
* 厳格release
* 月次release
* 承認工程が多い開発

## 向いていないケース

* 小規模開発
* 高頻度deploy
* スタートアップ系開発

# 5. GitHub Flow

GitHub Flow は、
GitHub中心文化に合わせたシンプル運用です。

## 基本構成

```plaintext
main
 └ feature/*
```

feature branch 作成後、
PR を通して main へ merge します。

## 基本フロー

```plaintext
branch作成
→ commit
→ push
→ Pull Request
→ review
→ merge
→ deploy
```

## 特徴

* シンプル
* branch数少ない
* 高速開発向き
* CI/CD相性良い

## 重要な前提

GitHub Flow では、
main が常に deploy可能状態であることが重要です。

そのため、

* CI必須
* review文化
* 自動テスト
* main保護

が重要になります。

## 向いているケース

* Web開発
* SaaS
* 高頻度deploy
* 少人数〜中規模

# 6. trunk based development

trunk based development は、
長期branchを極力減らす運用です。

trunk は main 相当の意味です。

## イメージ

```plaintext
main(trunk)
 ├ 小さい変更
 ├ 小さい変更
 ├ 小さい変更
 └ 小さい変更
```

## 特徴

* branch寿命が短い
* merge衝突減少
* deploy高速
* CI/CD前提

feature branch を長期間維持しないことが重要です。

## よく使う技術

* Feature Flag
* 小さいcommit
* 自動テスト
* 自動deploy

## 向いているケース

* DevOps文化
* CI/CD成熟組織
* 高頻度deploy
* マイクロサービス

## 注意点

CI/CD未成熟状態で導入すると、
main が壊れやすくなる場合があります。

# 7. fork運用

fork運用は、
元Repositoryを複製して開発する方式です。

OSS開発でよく使われます。

## イメージ

```plaintext
公式Repository
    ↓ fork
自分Repository
    ↓ PR
公式へ提案
```

## 特徴

* 本Repositoryを直接触らない
* 権限制御しやすい
* OSS向け

## 向いているケース

* OSS参加
* 外部Contributor受け入れ
* 権限制限が厳しい組織

## 注意点

fork同期を忘れると、
本Repositoryとの差異が大きくなる場合があります。

```plaintext
upstream取得忘れ
↓
古い状態で開発
↓
大量conflict
```

# 8. monorepo

monorepo は、
複数プロジェクトを1つのRepositoryで管理する方式です。

## イメージ

```plaintext
repository
 ├ frontend
 ├ backend
 ├ infra
 └ shared-lib
```

## 特徴

* 共通管理しやすい
* version同期しやすい
* 横断修正しやすい

## 向いているケース

* マイクロサービス群
* 共通ライブラリ多数
* 大規模組織

## 注意点

巨大化すると、

* clone重い
* CI時間増加
* 権限分離難化

などが発生します。

# 9. submodule

submodule は、
別Repositoryを内部参照する仕組みです。

## イメージ

```plaintext
main-repository
 └ shared-lib(submodule)
```

## 利点

* Repository分離維持
* 独立version管理可能
* 共通ライブラリ共有可能

## 注意点

submodule は運用難易度が高めです。

例えば、

```plaintext
submodule更新忘れ
↓
動作差異発生
```

```plaintext
clone後
↓
submodule初期化忘れ
↓
ファイル不足
```

などが発生しやすいです。

## よく使う操作

```bash
git submodule update --init --recursive
```

# 10. CI/CD連携

現代Git運用では、
CI/CD と GitHub連携が非常に重要です。

## CIとは

CI( Continuous Integration ) は、
変更を継続的に統合・検証する仕組みです。

## CDとは

CD( Continuous Delivery / Deployment ) は、
継続的deployを行う仕組みです。

## イメージ

```plaintext
push
↓
PR作成
↓
自動test
↓
review
↓
merge
↓
自動deploy
```

## よくある自動化

| 自動化内容         | 例                |
| ------------- | ---------------- |
| lint          | ESLint           |
| test          | Jest             |
| build         | TypeScript build |
| deploy        | AWS/GCP deploy   |
| security scan | 脆弱性検査            |

## Git運用との関係

CI/CDがあることで、

* main保護
* deploy品質安定
* review補助
* 人的ミス削減

が実現しやすくなります。
