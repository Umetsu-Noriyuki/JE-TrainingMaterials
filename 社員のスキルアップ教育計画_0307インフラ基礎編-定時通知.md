# 定時処理と通知の自動化 (AWS EventBridge と SNS)

企業のシステム運用や業務プロセスの効率化には、決まった時間やイベントをトリガーにして自動処理や通知を行う仕組みが不可欠です。  
この章では、AWSの代表的な定時処理サービス **EventBridge**( イベント連携 )と、通知・メッセージ配送を担う **SNS**( Simple Notification Service )を解説します。  

これらのサービスを使いこなすことで、複雑なタスクの自動化やスタッフへの迅速な情報伝達が可能になり、運用負荷の軽減とレスポンスの高速化が実現します。

## 1. AWS EventBridge とは？ ～イベントをきっかけに自動で動く仕組み～

**EventBridge (イベントブリッジ)** は、AWS内外のイベントを受け取って、その情報を元に設定した処理( ターゲット )を自動実行するイベント連携サービスです。  

### 特徴
- AWSサービスタイプのイベント( EC2の起動やS3のファイル追加など )をトリガーにできる
- 定時スケジュール( Cron式 )でもイベント発生を設定可能
- 複数のサービスをつなげて自動化が容易
- サーバーレスで運用負荷が少ない

### 実務での利用例
- 毎朝のデータバックアップ処理を自動化する  
- 新ファイルアップロード時に処理を起動  
- システムログ異常時にアラート通知  

### イメージ図
```plaintext
[イベント発生源]--→[EventBridge (ルール)]--→[処理( Lambda / SQS / SNS など )]
       │                       │
  EC2起動、                  条件に合うイベントを抽出・転送
  定期時間など

```

## 2. EventBridge の基本構成と使い方

| 要素 | 説明 | 実例 |
|---|---|---|
| イベントソース | イベントの発生元。AWSサービスやカスタムイベントなど | EC2の状態変更、スケジュール時間 |
| ルール(Rule) | イベントの受信条件とターゲットを定義 | 毎日午前9時に起動、EC2が停止したら通知 |
| ターゲット | イベントを受けて動作するサービス | Lambda、SNS、SQS、Step Functions |

**操作の流れ**

1. イベントソースを指定  
2. ルールで条件を設定( 例：Cronで毎日9時に発火 )  
3. ターゲットに処理を設定( 例：Lambda関数呼び出し )  

初心者はまずスケジュールイベントで定時処理を作成する練習がおすすめです。

## 3. AWS SNS とは？ ～多彩な通知を誰にでも素早く届けるメッセージ配信サービス～

**SNS (Simple Notification Service)** は、メッセージの配信と通知を行うサービスです。  

### 特徴
- メール、SMS、モバイルプッシュ通知、HTTPなど多様な配信先に対応  
- トピック単位で購読者を管理可能  
- 他サービスと連携してリアルタイム通知が可能  
- スケーラブルかつ信頼性の高い配信基盤  

### 実務での利用例
- システム障害発生時の管理者へのメール通知  
- ユーザーへの重要なお知らせのSMS送信  
- 開発チームへのSlack通知(Webhook連携)  

### 利用イメージ
```plaintext
[発信元サービス(Lambda/EventBridgeなど)]
            │
            ▼
         [SNS Topic]
       ┌─────────────┐
       │ メール｜SMS｜HTTP POST ...│
       └─────────────┘
            │
     [複数購読者へ同時配信]
```

## 4. EventBridge と SNS の連携例
EventBridgeのルールで発生したイベントをSNSのトピックに送信することで、イベント発生時にスタッフへ自動通知が可能です。

### 具体例：EC2の異常停止時にメール通知を送る流れ
1. EventBridgeで「EC2停止イベント」を監視するルールを作成  
2. ルールのターゲットにSNSトピックを指定  
3. SNSトピックに管理者のメールを登録  
4. EC2が停止すると、EventBridgeがSNSに通知を送信  
5. SNSが登録者にメールを送信

## 5. 実務でのポイントと注意点

- **イベント設計はシンプルに**  
  複雑なルールや過剰なターゲットはパフォーマンス低下や運用負荷の原因に。段階的に拡張しましょう。

- **通知の受け取り手の管理**  
  SNSの購読者は定期的に見直して、必要な人にだけ通知が届くよう管理しましょう。

- **コスト管理**  
  イベント数と通知数により変動するため、多数の頻繁な通知を送らない工夫が必要です。

- **IAM権限の最小化**  
  EventBridgeのルールやSNSトピックに対するアクセス権限は最小限に設定。誤操作や情報漏洩リスクを減らします。

## 6. まとめ

- **EventBridge**は「イベント発生→自動処理開始」のキッカケを作るサービス  
- **SNS**は「情報や通知を複数へ・多様に」素早く届けるメッセージ基盤  
- 両者の連携により、運用の自動化と迅速な通知が可能  
- 実務では「安定・シンプルな設計」と「コスト・権限管理」が重要

## 7. 重要用語説明

| キーワード | 説明 |
|------------|-------|
| EventBridge (イベントブリッジ) | AWSのイベント連携サービス。イベント発生をトリガーに自動処理を実行 |
| SNS<br>Simple Notification Service | AWSのメッセージ配信サービス。メール・SMS等へ通知可能 |
| Rule( ルール ) | EventBridgeでイベントを検出し、動作を指定する設定 |
| Target( ターゲット ) | イベントに応じて処理されるAWSサービスやエンドポイント |
| Topic( トピック ) | SNSでメッセージ配信先をグループ管理する単位 |
| Cron式 | スケジュール設定で用いる日時指定フォーマット |
| IAM( 管理コンソール )<br>Identity and Access Management | AWSリソースの権限管理サービス |
