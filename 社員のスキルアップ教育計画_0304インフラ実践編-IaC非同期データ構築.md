# IaC非同期データ構築<br>( SQS、S3、Firehose、Glue、IAM Role )

## 1. 非同期・データ連携構成とは？

非同期・データ連携構成とは、**「即時応答が必要な処理」と「裏側でじっくり実行する処理」を分離する構成**です。  
例えば、Webサイトで注文ボタンを押した際、画面には「受付完了」と即座に表示しつつ、裏側で「在庫更新」や「メール送信」を順番に処理する構成がこれに当たります。  

AWSでは主に以下を組み合わせます。
 - **SQS**：データを一時的に溜める「待機列（キュー）」
 - **S3**：ログやCSV、画像などを保存する「ストレージ」
 - **Firehose**：大量データを自動で配送する「運び屋」
 - **Glue**：データを加工・整形する「工場（ETL）」

### どんな場面で使われる？
以下のような “裏側で時間がかかる処理” を非同期化することで、ユーザー体験とシステム安定性が大きく向上します。

 - 注文受付後に、裏側でメール送信
 - 画像アップロード後に、非同期で画像変換
 - アクセスログを蓄積して後で集計
 - CSV を保存して後で Glue で加工し分析基盤へ連携

```plaintext
利用者操作
   ↓
受付完了（即時応答）
   ↓
裏側で順次処理（非同期）
```

👉 **「ユーザーにはすぐ返す」＋「裏側でしっかり処理」** が非同期構成の本質です。

## 2. YAMLファイルの基本構造
非同期構成を IaC で管理する場合も、CloudFormation の基本構造は共通です。1つのテンプレートで入口から出口までのリソースを管理します。

```plaintext
Root
├─ AWSTemplateFormatVersion
├─ Parameters（環境差分の入力値）
├─ Resources（非同期構成の中心）
│   ├─ AppApi（API Gateway：受付の入り口）
│   ├─ AppLambda（受付用Lambda：SQSへデータを送る）
│   ├─ AppLambdaPermission（API Gateway からの実行許可）★追加
│   ├─ AppQueue（SQS：メッセージの待機列）
│   ├─ WorkerLambda（処理用Lambda：キューを処理する）
│   ├─ QueueEventSource（SQS と WorkerLambda の紐付け）
│   ├─ DataBucket（S3：データの保存先）
│   ├─ FirehoseDelivery（Firehose：ログ等の自動配送）
│   ├─ GlueJob（Glue：ETL加工処理）
│   └─ AppRole（各サービス用の IAM Role）
└─ Outputs（保存先やキューURL）
```

### 各項目の役割
 - Parameters  
 dev / prod の切り替えや、キューの保持期間などの設定値を外部入力として管理します。
 - Resources  
 SQS、S3、Firehose、Glue、IAM Role など、非同期・データ連携構成の中心となるリソースを定義します。
 - Outputs  
 作成した S3 バケット名や SQS キューの URL を、他のテンプレートや出力確認用として外部へ出力する際に使用します。

👉 この構造を理解しておくと、複数のサービスが連携する複雑な IaC も整理して記述できるようになります。

## 3. SQS を IaC で定義する（非同期キュー）

SQS は、処理すべきメッセージを一時的に保持するサービスです。  
SQS（Simple Queue Service）は、処理待ちデータを一時保存する「待機列」として機能します。システムに急激な負荷がかかっても、このキューが「緩衝材」となってデータの消失を防ぎます。

例：

```yaml
Resources:
  # 異常データを受け取るための隔離用キュー
  AppDeadLetterQueue:
    Type: AWS::SQS::Queue
    Properties:
      QueueName: app-dlq-dev

  # メインのキュー
  AppQueue:
    Type: AWS::SQS::Queue
    Properties:
      QueueName: app-queue-dev
      VisibilityTimeout: 30
      RedrivePolicy: # リトライと隔離の設定
        deadLetterTargetArn: !GetAtt AppDeadLetterQueue.Arn
        maxReceiveCount: 3 # 3回失敗したらDLQへ移動
```
### 可視性タイムアウト（VisibilityTimeout）の重要性
あるWorker（キューから仕事を受け取って実行するプログラム） がメッセージを処理中、他のWorkerからそのメッセージを一時的に見えなくする設定です。
 - **なぜ重要なのか？（二重処理の防止）**:もし処理が終わる前にこの時間が切れてしまうと、メッセージが「待機列」に再び現れてしまいます。すると、別の Worker が「まだ誰もやっていない仕事だ」と勘違いして同じ処理を始めてしまい、データの重複や二重送信が発生してしまいます。
 - **設計の鉄則**：後続の Worker Lambda （実際にデータ処理を行うプログラム）のタイムアウト時間よりも必ず長く設定します。
    - **理由**: Worker Lambda が「30秒かけて処理する」設定なのに、SQS が「20秒で次を出す」設定だと、処理が終わる前に次の担当者が現れてしまうからです。

### 利用例
実務では、以下のような「時間のかかる処理」や「失敗の可能性がある処理」を非同期化するために利用します。
Lambda のタイムアウト時間よりも必ず長く設定（例：Lambdaが30秒ならSQSは120秒など）します。
 - **注文データの後続処理**: 決済完了後、在庫更新や配送指示を裏側で実行する
 - **メール送信**: APIのレスポンスを待たせずに、大量の通知メールを順次送信する
 - **バッチ処理**: 大量のデータ更新を、一定のペースで確実に処理する
 - **リトライ処理**: 外部APIとの連携などで一時的にエラーが出た際、自動的に再試行させる

### デッドレターキュー（DLQ）の検討
実務で SQS を導入する場合、**「何度リトライしても処理できない異常なデータ」**への対策が必要です。
 - **DLQ とは**: 指定回数リトライしても失敗したメッセージを、隔離して保存するための別のキューです。
 - **メリット**: 異常なデータが原因でキューが詰まる（無限リトライ）のを防ぎ、後から手動で調査・修正ができるようになります。

👉 混雑時の緩衝材として非常に有効です。

## 4. S3 を IaC で定義する（データ保存）

S3 は、あらゆる形式のデータを安全に格納するオブジェクトストレージサービスです。  
S3（Simple Storage Service）は、CSV、画像、ログなど、あらゆるデータを保存する非同期構成の「データ蓄積基盤」です。IaCではバケットの作成だけでなく、アクセス制限や保存期間の管理を行います。

例：

```yaml
Resources:
  DataBucket:
    Type: AWS::S3::Bucket
    Properties:
      BucketName: app-data-bucket-dev
      PublicAccessBlockConfiguration:
        BlockPublicAcls: true
        BlockPublicPolicy: true
```

### 利用例

- CSVアップロード
- アクセスログ保存
- バックアップ
- Glue の入力データ

👉 非同期構成の中心となる保存基盤です。

## 5. Firehose を IaC で定義する（データ配送）

Firehose は、リアルタイムのストリーミングデータを他のサービスへ自動配送するサービスです。  
Firehose を使うことで、大量のデータを逐次 S3 や分析基盤へ「自動で」配送できます。プログラムを書かずに「溜めてから送る」というバッファリング処理を実現できるのが最大の特徴です。  

例：

- アプリログの収集: サーバーから送られる大量の実行ログを S3 に集約する
- イベントデータの蓄積: ユーザーの操作イベントを Redshift（分析基盤）へ送る
- バックアップ: リアルタイムに発生する重要データのコピーを保存する

```plaintext id="v8p3ua"
 大量ログ（断続的に発生）
      ↓
 [ Firehose ] ← ここで「まとめて」くれる
      ↓
 S3 / 分析基盤（整理されたファイルとして保存）
```

```yaml
Resources:
  FirehoseDelivery:
    Type: AWS::KinesisFirehose::DeliveryStream
    Properties:
      DeliveryStreamType: DirectPut
      S3DestinationConfiguration:
        BucketARN: !GetAtt DataBucket.Arn
        RoleARN: !GetAtt FirehoseRole.Arn
        BufferingHints:
          IntervalInSeconds: 60 # 60秒分溜めてから送る
          SizeInMBs: 5 # または5MB溜まったら送る
```

### バッファリング設定の重要性
実務で Firehose を定義する際、BufferingHints の設定は非常に重要です。

- **コストの最適化**: S3 は「書き込み回数」でも料金が発生します。1件ずつ書き込むのではなく、Firehose でまとめてから書き出すことで、S3 の費用を大幅に抑えることができます。
- **後続処理の効率化**: 1KB のファイルが数万個あるよりも、5MB 程度のファイルが少数ある方が、後続の Glue や Athena での分析スピードが格段に速くなります。

👉 「都度送る」のではなく「効率よくまとめて送る」 設定を入れるのが IaC 定義のコツです。

## 6. Glue を IaC で定義する（ETL処理）

Glue は、データの抽出・変換・書き出し（ETL）を自動化するフルマネージドなデータ加工サービスです。  
Glue を利用することで、S3 に保存された未加工のデータを、分析しやすい形式（CSVからParquetなど）へ自動変換したり、不要な情報を削除したりする「工場」のような役割を担います。

ETLとは：  
Glue の主な役割である ETL は、以下の3つの頭文字を取ったものです。初心者の方は「データを料理する工程」と考えるとイメージしやすくなります。

- Extract( 抽出 ):  
S3 などの貯蔵庫から、加工前の「生のデータ」を取り出す工程。（食材を冷蔵庫から出す）
- Transform( 変換 ):  
データの形式を変えたり、特定の列を削除したり、計算したりする工程。（食材を洗う、切る、味付けする）
- Load( 格納 ):  
加工が終わったデータを、分析用バケットやデータベースへ保存する工程。（料理をお皿に盛り付ける）

### 利用例

- フォーマット変換: 大量の CSV ファイルを、高速に検索できる Parquet 形式に変換する
- データのクレンジング: ログに含まれる個人情報（電話番号など）を削除して、分析用バケットへ移す
- システム間連携: Aシステムのデータ形式を、Bシステムが読み込める形式に整えて転送する

### 実務でのデータ処理イメージ

```plaintext
S3元データ
  ↓
Glue
  ↓
加工済みデータ
```

```yaml
Resources:
  GlueJob:
    Type: AWS::Glue::Job
    Properties:
      Name: app-data-etl-job
      Role: !GetAtt GlueRole.Arn # 加工に必要な権限
      Command:
        Name: glueetl # ETLジョブであることを指定
        ScriptLocation: !Sub s3://${DataBucket}/scripts/transform.py
      DefaultArguments:
        "--job-language": "python"
      MaxCapacity: 2.0 # 処理能力（ワーカー数）の指定
```

### コストとパフォーマンスのバランス
Glue は非常に強力ですが、設定次第でコストが変動します。IaC で定義する際に意識すべきポイントは以下の通りです。

 - MaxCapacity（ワーカー数）の調整:  
 まずは最小の「2」から始め、処理速度が不足する場合に増やすのが実務的なアプローチです。
 - タイムアウト設定:  
 Glue はデフォルトで長時間動いてしまう設定になっていることがあります。IaC で Timeout を適切に設定し、プログラムのバグによる予期せぬ課金を防ぐのが「プロの定義」です。

👉  Glue（ETL）は、バラバラな「生のデータ」を、価値のある「情報」へと磨き上げるための重要な橋渡し役です。

## 7. IAM Role（サービス連携権限）

IAM Role は、AWSのリソースに対して「他のサービスを操作する権限」を安全に貸し出すための仕組みです。  
非同期構成では、SQSからメッセージを取り出す、S3にファイルを書き込む、Glueでデータを読み込むといった「サービス間の連携」が頻繁に発生します。アクセスキーなどを直接プログラムに書き込むのではなく、Role（役割）を定義してリソースに割り当てるのがクラウド運用の鉄則です。

### 「IAM Role」とは？
「特定の作業をするための 『権限セットが入った通行証』 」とイメージしてください。
 - **Firehose → S3保存**: Firehoseという「運び屋」に、S3という「倉庫」へ荷物を置く許可を与える。
 - **Glue → S3読取**: Glueという「工場」に、S3から「原材料」を取り出す許可を与える。
 - **Lambda → SQS受信**: Lambdaという「作業員」に、SQSという「待機列」から仕事を取ってくる許可を与える。

例：

 ```plaintext
 [ 各サービス ]
( Lambda / Glue / Firehose )
      ↓
[ IAM Role ]
( 一時的な通行証 )
      ↓
[ 操作対象リソース ]
( SQS / S3 / CloudWatch )
```

YAML例
```yaml
Resources:
  AppGlueRole:
    Type: AWS::IAM::Role
    Properties:
      # 「誰が」このRoleを使えるか（信頼関係）
      AssumeRolePolicyDocument:
        Version: "2012-10-17"
        Statement:
          - Effect: Allow
            Principal:
              Service: glue.amazonaws.com
            Action: sts:AssumeRole
      # 「何が」できるか（権限の中身）
      ManagedPolicyArns:
        - arn:aws:iam::aws:policy/service-role/AWSGlueServiceRole
```
IaCでRoleを定義する際、必ず登場する AssumeRolePolicyDocument は、**「このRoleを誰（どのサービス）に使わせてよいか」**を定義する重要な設定です。
 - **Principal（プリンシパル）**: ここに lambda.amazonaws.com と書けばLambda用、glue.amazonaws.com と書けばGlue用のRoleになります。
 - **セキュリティの重要性**: もしここを間違えると、権限の内容（Policy）が正しくてもサービスが動かないため、実務で最もトラブルが起きやすいポイントの一つです。

👉 IAM Role は、サービス同士が「安全に・信頼して」連携するための鍵となる要素です。

## 8.サービス間連携と Permission（許可）の設定
AWSリソース同士が互いに通信・操作するための「許可」と「接続」の設定です。  
非同期構成では、サービス同士が「裏側で」連携するため、呼び出し許可の使い分けが重要になります。特に「誰が誰を叩くのか（プッシュ）」か「誰かが呼びに来るのを待つのか（プル）」の違いを意識する必要があります。  
まずは、この章で扱う2つの役割（Lambda）を整理しましょう。

- 受付用 Lambda (AppLambda):  
API Gateway から「叩かれて」動き、データを SQS へ放り込む「受付窓口」です。

- Worker Lambda (WorkerLambda):  
SQS に溜まったデータを「運んできてもらって」動く、実務担当の Lambda です。

### 「プッシュ型」と「プル型」の使い分け

- **AppLambdaPermission（プッシュ型）**：  
API Gateway が「受付用 Lambda」を直接呼び出すための許可です。
- **QueueEventSource（プル型）**：  
SQS のメッセージを検知して「Worker Lambda」を自動起動させる紐付け設定（EventSourceMapping）です。

### 「プッシュ型」と「プル型」の違い
動作のきっかけ（トリガー）の仕組みは大きく分けて2種類あります。
- プッシュ型  
API Gateway が Lambda を「直接叩く」イメージ。呼び出しには Lambda::Permission が必要。
- プル型  
SQS に溜まったデータを Lambda が「自分から見に行く（取りに行く）」イメージ。この接続路を作るのが Lambda::EventSourceMapping です。

  動作イメージ
  ```plaintext
  [ SQS：メッセージ到着 ]
        ↓
  [ EventSourceMapping ]
  ( データの見張り番 )
        ↓
  [ Worker Lambda ]
  ( 処理の実行 )
  ```

### SQS と Lambda の紐付け定義（EventSourceMapping）
```yaml
Resources:
  QueueEventSource:
    Type: AWS::Lambda::EventSourceMapping
    Properties:
      EventSourceArn: !GetAtt AppQueue.Arn # 監視するキューの場所
      FunctionName: !Ref WorkerLambda     # 起動する関数の名前
      BatchSize: 10 # 1度に最大10件まとめて処理する
```

実務で EventSourceMapping を定義する際、BatchSize の設定はパフォーマンスに直結します。
- **1 に設定した場合**: メッセージが届くたびに Lambda が動くため、リアルタイム性は高いですが、Lambda の起動回数（コスト）が増えます。
- **10 以上に設定した場合**: ある程度データが溜まってから Lambda を動かすため、処理効率が上がりコストを抑えられます。

👉 「権限（Role）」で道を通し、「接続（Mapping/Permission）」で動線を繋ぐことで、初めて非同期の連携が動き出します。

## 9.よくある非同期データの構成例
ここでは、データの入り口から加工・保存までの全体像を整理します。

### データの流れ（同期から非同期へ）
```plaintext
[ API利用者 ]
      ↓ (1) リクエスト
[ API Gateway ]
      ↓
[ 受付用 Lambda ]
      ├─ (2) 即時保存（S3 / DynamoDB）
      └─ (3) キューへ投入（SQS） ◀── ここから非同期開始！
            ↓
      [ Worker Lambda ]
            ↓ (4) 重い処理・データ加工
      [ Firehose / Glue ]
```

### CloudFormation で管理する「リソースの一覧」
```plaintext
Resources（システム全体の中心）
│
├─── 【入口：サーバレス層】
│   ├─ AppApi（API Gateway）
│   ├─ AppLambda（受付用：SQSへメッセージを送る）
│   └─ AppLambdaPermission（API Gateway からの実行許可）
│
├─── 【中継：非同期・配送層】
│   ├─ AppQueue（SQS：メインの待機列）
│   ├─ AppDeadLetterQueue（SQS：隔離用キュー）
│   └─ FirehoseDelivery（Firehose：大量データの自動配送）
│
└─── 【出口：加工・保存層】
    ├─ WorkerLambda（処理用：SQSからデータを受け取って動く）
    ├─ QueueEventSource（SQS と Worker の紐付け設定）
    ├─ DataBucket（S3：最終的な保存先）
    └─ GlueJob（Glue：データの磨き上げ・変換）
```

👉 「どこまでがユーザーを待たせる処理（同期）」で「どこからが裏側でやる処理（非同期）」かを切り分けて設計するのが、IaC構築の第一歩です。

## 10. よくある失敗例

| 失敗 | 原因 |
| --- | --- |
| メッセージが滞留する | Worker Lambda の同時実行不足や処理遅延 |
| 保存失敗 | IAM Role の S3 書き込み権限不足 |
| データが重複する | 可視性タイムアウトが処理時間より短い |
| 本番と開発のデータ混在 | バケット名が環境変数で動的に設定されていない |

### 11. まとめ
| SQS はシステムの「安定」を保つ緩衝材。
| DLQ を設定することで、異常系データの処理を安全に分離できる。
| AppLambdaPermission は、非同期であっても「入り口」を動かすために必須。
| Firehose/Glue は、大量データを「効率よく運び、整える」ための道具。
| IaC化 により、これら複雑な連携設定をミスなく一発で再現可能にする。

## 12. 重要用語説明

| 用語 | 説明 |
|---|---|
| CloudFormation<br>( クラウドフォーメーション ) | AWSリソースを自動構築するためのツール |
| SQS( メッセージキュー )<br>Amazon Simple Queue Service | データを一時保存し順番に処理する仕組み |
| DLQ( 隔離用キュー )<br>Dead Letter Queue | 処理に失敗し続けた異常データを、調査のために隔離しておく専用のキュー。 |
| Visibility Timeout( 可視性タイムアウト ) | メッセージを処理している間、他の作業員から見えないようにする設定。二重処理を防ぐ鍵 |
| S3( オブジェクトストレージ )<br>Amazon Simple Storage Service | ファイル・ログ保存の基盤 |
| Firehose( データ配送 ) | 大量データを S3 や分析基盤へ自動配送 |
| Buffering Hints( バッファリングヒント ) | Firehoseで「どれくらいデータが溜まったらS3へ書き出すか」の設定値 |
| Glue( ETLサービス ) | データ抽出・変換・格納を行うサービス |
| IAM Role( 権限ロール )<br>Identity and Access Management Role | AWSサービスへ権限を委任する仕組み |
| ETL( 抽出変換格納 )<br>Extract Transform Load | データを抽出し加工して保存する処理 |
| AssumeRole( アシューム・ロール ) | Roleを定義する際の「誰（どのサービス）がこの役割を使えるか」という信頼関係の設定 |
| EventSourceMapping<br>（イベントソースマッピング） | SQS や Kinesis などのイベントを Lambda に紐付ける設定。キューのメッセージを自動で Lambda に渡す仕組み。 |
| BatchSize（バッチサイズ） | Lambda が一度に処理する SQS メッセージ数。処理速度やコストに影響する。 |
| DirectPut（ダイレクトプット） | Firehose に直接データを送信するモード。アプリケーションから Firehose に直接書き込む方式。 |
| Worker( 処理担当 ) | キューから取得して処理する担当 |
