# IaC非同期データ構築<br>( SQS、S3、Firehose、Glue、IAM Role )

## 1. 非同期・データ連携構成とは？

非同期・データ連携構成とは、  
**「すぐ返す処理」と「後で順番に処理する部分」を分離する構成**です。

また、業務データやログデータを蓄積し、後から加工・分析しやすくします。  
これにより、画面応答を速くしつつ、大量処理や時間のかかる処理を安全に実行できます。

### どんな場面で使われる？
以下のような “裏側で時間がかかる処理” を非同期化することで、ユーザー体験とシステム安定性が大きく向上します。

 - 注文受付後に、裏側でメール送信
 - 画像アップロード後に、非同期で画像変換
 - アクセスログを蓄積して後で集計
 - CSV を保存して後で Glue で加工し分析基盤へ連携

```plaintext
利用者操作
   ↓
受付完了（即時応答）
   ↓
裏側で順次処理（非同期）
```

👉 **「ユーザーにはすぐ返す」＋「裏側でしっかり処理」** が非同期構成の本質です。

## 2. YAMLファイルの基本構造
非同期構成を IaC で管理する場合も、CloudFormation の基本構造は同じです。

```plaintext
Root
├─ AWSTemplateFormatVersion
├─ Parameters（環境差分）
├─ Resources（非同期構成の中心）
│   ├─ Queue（SQS）
│   ├─ DataBucket（S3）
│   ├─ FirehoseDelivery（Firehose）
│   ├─ GlueJob（Glue）
│   └─ Role（各サービスの権限）
└─ Outputs（保存先やキューURL）
```
👉 0302 と同じ構造で、複数サービスを 1 テンプレートで管理できます。

## 3. SQS を IaC で定義する（非同期キュー）

SQS は「後で処理するデータ」を一時保存するキューです。

```yaml
Resources:
  AppQueue:
    Type: AWS::SQS::Queue
```

### 利用例

 - 注文データの後続処理
 - メール送信
 - バッチ処理
 - リトライ処理

👉 混雑時の緩衝材として非常に有効です。

## 4. S3 を IaC で定義する（データ保存）

S3 はログ・CSV・画像などを保存するストレージです。

```yaml
Resources:
  DataBucket:
    Type: AWS::S3::Bucket
    Properties:
      BucketName: app-data-dev
```

### 利用例

- CSVアップロード
- アクセスログ保存
- バックアップ
- Glue の入力データ

👉 非同期構成の中心となる保存基盤です。

## 5. Firehose を IaC で定義する（データ配送）

Firehose は大量データを自動で S3 や分析基盤へ配送します。

例：

* アプリログ → S3
* イベントデータ → Redshift
* 分析データ収集

```plaintext id="v8p3ua"
大量ログ
 ↓
Firehose
 ↓
S3 / 分析基盤
```

### 特徴

* 自動バッファ保存
* 大量データ向き
* 運用負荷が低い

👉 ログ収集や分析基盤連携でよく使われます。

## 6. Glue を IaC で定義する（ETL処理）

Glue はデータ加工・変換を行うサービスです。

ETLとは：

* Extract( 抽出 )
* Transform( 変換 )
* Load( 格納 )

### 利用例

* CSV整形
* 日付変換
* データ結合
* 分析用形式へ変換

```plaintext
S3元データ
  ↓
Glue
  ↓
加工済みデータ
```
👉 \*\*非同期構成の「加工担当」\*\*です。

## 7. IAM Role（サービス連携権限）

各サービスが連携するには Role が必要です。

例：

* Firehose → S3保存
* Glue → S3読取
* Lambda → SQS受信

```yaml
Resources:
  AppGlueRole:
    Type: AWS::IAM::Role
    Properties:
      AssumeRolePolicyDocument:
        Version: "2012-10-17"
        Statement:
          - Effect: Allow
            Principal:
              Service: glue.amazonaws.com
            Action: sts:AssumeRole
```

👉 AccessKey 直書きではなく Role 利用が基本です。

## 8. よくある非同期構成（実務例）

## 8-1 注文処理

```plaintext
Web注文
   ↓（即時応答）
SQS（注文キュー）
   ↓
Worker Lambda
   ├─ 在庫更新
   └─ メール送信
```
### YAML例（最小構成）
```yaml
Resources:
  OrderQueue:
    Type: AWS::SQS::Queue

  WorkerLambda:
    Type: AWS::Lambda::Function
    Properties:
      Runtime: python3.12
      Handler: index.lambda_handler
      Role: !GetAtt WorkerLambdaRole.Arn
      Code:
        ZipFile: |
          def lambda_handler(event, context):
              print("process order")

  # Lambda が SQS を受信するためのイベント設定
  QueueEventSource:
    Type: AWS::Lambda::EventSourceMapping
    Properties:
      EventSourceArn: !GetAtt OrderQueue.Arn
      FunctionName: !Ref WorkerLambda
      BatchSize: 10
```

👉 混雑時でも安定し、裏側処理を安全に分離できる構成です。

## 8-2 ログ分析

```plaintext
アプリログ
   ↓
Firehose
   ↓
S3（ログ保存）
   ↓
Glue（加工）
   ↓
分析基盤
```

### YAML例（最小構成）
```yaml
Resources:
  LogBucket:
    Type: AWS::S3::Bucket

  FirehoseDelivery:
    Type: AWS::KinesisFirehose::DeliveryStream
    Properties:
      DeliveryStreamType: DirectPut
      S3DestinationConfiguration:
        BucketARN: !GetAtt LogBucket.Arn
        RoleARN: !GetAtt FirehoseRole.Arn

  FirehoseRole:
    Type: AWS::IAM::Role
    Properties:
      AssumeRolePolicyDocument:
        Version: "2012-10-17"
        Statement:
          - Effect: Allow
            Principal:
              Service: firehose.amazonaws.com
            Action: sts:AssumeRole

```

👉 大量ログを自動で蓄積し、後で加工・分析できる構成です。

## 8-3 CSVファイル連携

```plaintext
CSVアップロード
   ↓
S3
   ↓
Glue Job（加工）
   ↓
別システムへ連携
```

### YAML例（最小構成）
```yaml
Resources:
  DataBucket:
    Type: AWS::S3::Bucket

  GlueJob:
    Type: AWS::Glue::Job
    Properties:
      Role: !GetAtt GlueRole.Arn
      Command:
        Name: glueetl
        ScriptLocation: !Sub s3://${DataBucket}/scripts/job.py
```

👉 ファイル連携の定番構成で、業務システムとの接続に使われます。

## 9. よくある非同期構成（代表的な構成）

非同期構成では、**即時応答が必要な処理**と、
**裏側で順次実行すべき処理**を分離することで、
システムの安定性とスケーラビリティを高めます。  
ここでは、実務で最も利用される **「API → SQS → Worker → S3/Glue」** の構成を例に、
全体像と IaC のポイントを解説します。

### 10-1. 非同期構成の全体図
```plaintext
利用者
  ↓（即時応答）
受付API
  ↓（後続処理を依頼）
SQS
  ↓
Worker（Lambda）
  ├─ S3保存
  ├─ Glueジョブ起動
  └─ CloudWatch Logs
```
### この構成の特徴
 - API は **すぐに応答を返せる**
 - 時間のかかる処理は **SQS に積んで後で実行**
 - Worker Lambda が **順番に処理**
 - 結果を S3 に保存したり、Glue で加工したりできる
 - 障害時も **キューに残るためデータが失われにくい**

👉 0302 の**「API → Lambda → S3/DynamoDB」構成の非同期版**と考えると理解しやすいです。

### 9-2. この構成で必要な AWS リソース
| リソース | 役割 |
| --- | --- |
| SQS | 後続処理をためるキュー |
| Worker Lambda | キューを処理する実行担当 |
| EventSourceMapping | SQS → Lambda の紐付け |
| S3 | 結果保存・ログ保存 |
| Glue | データ加工 |
| IAM Role | 各サービスの権限管理 |
| CloudWatch Logs | Lambda のログ確認 |

👉 0302 の構成と同様に、各サービスの役割を明確に分離することが重要です。

### 9-3. YAML（最小構成例）
```yaml
AWSTemplateFormatVersion: '2010-09-09'

Parameters:
  env:
    Type: String
    Default: dev

Resources:

  # ----------------------------------------
  # SQS: 非同期キュー
  # ----------------------------------------
  AppQueue:
    Type: AWS::SQS::Queue
    Properties:
      QueueName: !Sub app-queue-${env}

  # ----------------------------------------
  # IAM Role: Worker Lambda の実行権限
  # ----------------------------------------
  WorkerLambdaRole:
    Type: AWS::IAM::Role
    Properties:
      RoleName: !Sub worker-lambda-role-${env}
      AssumeRolePolicyDocument:
        Version: '2012-10-17'
        Statement:
          - Effect: Allow
            Principal:
              Service: lambda.amazonaws.com
            Action: sts:AssumeRole
      Policies:
        - PolicyName: WorkerLambdaPolicy
          PolicyDocument:
            Version: '2012-10-17'
            Statement:
              - Effect: Allow
                Action:
                  - logs:CreateLogGroup
                  - logs:CreateLogStream
                  - logs:PutLogEvents
                Resource: "*"
              - Effect: Allow
                Action:
                  - s3:PutObject
                Resource: "*"

  # ----------------------------------------
  # Lambda: Worker（キュー処理担当）
  # ----------------------------------------
  WorkerLambda:
    Type: AWS::Lambda::Function
    Properties:
      FunctionName: !Sub worker-lambda-${env}
      Runtime: python3.12
      Handler: index.lambda_handler
      Role: !GetAtt WorkerLambdaRole.Arn
      Timeout: 30
      MemorySize: 256
      Code:
        ZipFile: |
          import json
          import boto3

          def lambda_handler(event, context):
              print("process message:", event)
              return {"statusCode": 200}

  # ----------------------------------------
  # EventSourceMapping: SQS → Lambda
  # ----------------------------------------
  QueueEventSource:
    Type: AWS::Lambda::EventSourceMapping
    Properties:
      EventSourceArn: !GetAtt AppQueue.Arn
      FunctionName: !Ref WorkerLambda
      BatchSize: 10
      Enabled: true

  # ----------------------------------------
  # S3: 結果保存
  # ----------------------------------------
  AppBucket:
    Type: AWS::S3::Bucket
    Properties:
      BucketName: !Sub app-bucket-${env}

Outputs:
  QueueUrl:
    Value: !Ref AppQueue
  WorkerLambdaName:
    Value: !Ref WorkerLambda
```

### 9-4. この構成でよくある注意点
|注意点 | 説明 |
| --- | --- |
| Worker が遅いと SQS が滞留 | Lambda の同時実行数や処理速度を調整する必要あり |
| IAM 権限不足で保存失敗 | S3 / Glue / Logs の権限を明示的に付与する |
| EventSourceMapping の設定漏れ | SQS → Lambda が動かない原因の 1 位 |
| Glue の処理が重い | メモリ・ワーカー数の調整が必要 |

## 10. 実行コマンド

### 構文確認

```bash
aws cloudformation validate-template \
--template-body file://template.yaml
```

### デプロイ

```bash id="w1n3dr"
aws cloudformation deploy \
--template-file template.yaml \
--stack-name async-stack \
--capabilities CAPABILITY_NAMED_IAM
```

### 削除

```bash id="c0m7ls"
aws cloudformation delete-stack \
--stack-name async-stack
```

## 11. よくある失敗例

| 失敗      | 原因       |
| ------- | -------- |
| メッセージ滞留 | Worker不足 |
| 保存失敗    | IAM権限不足  |
| データ欠損   | Glue変換ミス |
| 本番混在    | バケット分離不足 |

## 12. まとめ

- SQS は非同期処理の中心
- S3 は保存基盤
- Firehose は大量配送向け
- Glue は加工・変換担当
- IAM Role は安全な連携に必須
- IaC化で複雑な構成を安全管理できる

👉 非同期構成は **速い画面応答と強い裏側処理** を両立します。

## 13. 重要用語説明

| 用語 | 説明 |
|---|---|
| SQS( メッセージキュー )<br>Amazon Simple Queue Service | データを一時保存し順番に処理する仕組み |
| S3( オブジェクトストレージ )<br>Amazon Simple Storage Service | ファイル・ログ保存の基盤 |
| Firehose( データ配送 )<br>Amazon Data Firehose | 大量データを S3 や分析基盤へ自動配送 |
| Glue( ETLサービス )<br>AWS Glue | データ抽出・変換・格納を行うサービス |
| IAM Role( 権限ロール )<br>Identity and Access Management Role | AWSサービスへ権限を委任する仕組み |
| ETL( 抽出変換格納 )<br>Extract Transform Load | データを抽出し加工して保存する処理 |
| Queue( 待機列 )<br>Queue | 処理待ちデータを順番に並べる仕組み |
| Batch( 一括処理 )<br>Batch Processing | まとめて実行する処理方式 |
| EventSourceMapping<br>（イベントソースマッピング） | SQS や Kinesis などのイベントを Lambda に紐付ける設定。キューのメッセージを自動で Lambda に渡す仕組み。 |
| DeliveryStream（デリバリーストリーム） | Firehose がデータを受け取り、S3 や Redshift に配送するためのストリーム。 |
| BatchSize（バッチサイズ） | Lambda が一度に処理する SQS メッセージ数。処理速度やコストに影響する。 |
| DirectPut（ダイレクトプット） | Firehose に直接データを送信するモード。アプリケーションから Firehose に直接書き込む方式。 |
| Deploy( 反映 )<br>Deploy | コードや設定をAWS環境へ適用すること |
| Worker( 処理担当 )<br>Worker | キューから取得して処理する担当 |
