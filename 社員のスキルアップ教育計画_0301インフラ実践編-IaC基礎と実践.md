# IaC基礎と実践<br>( Infrastructure as Code、CloudFormation、YAML、Deploy )

## 1. IaC( Infrastructure as Code )とは？

IaC( Infrastructure as Code )とは、  
サーバー・ネットワーク・権限設定などのインフラ構成を、**コードとして管理する考え方**です。

従来は管理画面から手作業で構築していましたが、IaCでは設定内容をファイル化し、再現可能な形で管理します。

- **目的**：構築作業の標準化、自動化、再現性向上
- **効果**：設定ミス削減、レビュー可能、変更履歴管理
- **代表例**：CloudFormation、Terraform

```plaintext
[従来]
　管理画面で手作業設定

[IaC]
　コードを書く → 実行 → 同じ環境を再現
```

👉 **インフラを人の記憶ではなく、コードで管理するのがIaCです。**

## 2. AWS標準IaC - CloudFormation

CloudFormation は AWS が提供する IaC サービスです。
YAML または JSON でAWS構成を定義し、その内容をAWSへ反映できます。

- AWS公式サービスのため相性が良い
- 多くのAWSサービスに対応
- 変更差分を安全に適用可能
- Stack単位で管理できる

```plaintext
YAMLファイル
   ↓
CloudFormation
   ↓
VPC / S3 / Lambda / IAM など作成
```

## 3. CloudFormationテンプレート( YAML )について
CloudFormationでは、YAMLファイルにAWS構成を書き、環境を自動作成します。

### 3-1. 基本構造
テンプレートの中に、複数のAWSリソースをまとめて記述できます。  

```plaintext
Root( テンプレート全体 )
├─ AWSTemplateFormatVersion
├─ Parameters( 入力値 )
├─ Resources( 作成するAWSリソース )
│   ├─ 論理ID 1( AWSリソース 1: Type + Properties )
│   ├─ 論理ID 2( AWSリソース 2: Type + Properties )
│   ├─  ...
│   └─ 論理ID N( AWSリソース N: Type + Properties )
└─ Outputs( 作成結果 )
```
👉 Parameters は入力値、Resources は作成内容、Outputs は作成結果です。

以下は主要要素を含んだ最小例です。

```yaml
AWSTemplateFormatVersion: '2010-09-09'

Parameters:
  env:
    Type: String
    Default: dev

Resources:

  AppBucket:
    Type: AWS::S3::Bucket
    Properties:
      BucketName: !Sub app-bucket-${env}
      Tags:
        - Key: Env
          Value: !Ref env

  AppLambdaRole:
    Type: AWS::IAM::Role
    Properties:
      RoleName: !Sub app-lambda-role-${env}
      AssumeRolePolicyDocument:
        Version: '2012-10-17'
        Statement:
          - Effect: Allow
            Principal:
              Service: lambda.amazonaws.com
            Action: sts:AssumeRole
      Policies:
        - PolicyName: S3WritePolicy
          PolicyDocument:
            Version: '2012-10-17'
            Statement:
              - Effect: Allow
                Action:
                  - s3:PutObject
                Resource: !Sub arn:aws:s3:::app-bucket-${env}/*

  AppLambda:
    Type: AWS::Lambda::Function
    Properties:
      FunctionName: !Sub app-lambda-${env}
      Runtime: python3.11
      Handler: index.lambda_handler
      Role: !GetAtt AppLambdaRole.Arn
      Code:
        ZipFile: |
          # Lambda実行コード
          def lambda_handler(event, context):
              return {"statusCode": 200, "body": "ok"}

Outputs:
  BucketName:
    Value: !Ref AppBucket
```
👉 CloudFormationは「単一サービスの定義」ではなく「複数リソースの集合」です。  

#### 📌 上記YAMLのリソース対応表
| # | 論理ID | Type | 役割 | 物理ID |
|---|---|---|---|---|
| 1 | AppBucket | S3 Bucket | データ保存 | app-bucket-dev |
| 2 | AppLambdaRole | IAM Role | Lambdaへ権限付与 | app-lambda-role-dev |
| 3 | AppLambda | Lambda | Pythonコード実行 | app-lambda-dev |

- 論理ID( Logical ID ): テンプレート内で管理する名前
- 物理ID( Physical ID ): AWS上で実際に作成されるリソース名

#### 📌 ミニコラム　!Ref / !Sub / !GetAtt とは？
CloudFormation では、他の値を参照したり、文字列に値を埋め込む専用記法があります。

| 記法 | 用途 | 例 | 結果(env=dev) |
|---|---|---|---|
| !Ref | 値を参照する | !Ref env | dev |
| !Sub | 文字列内に値を埋め込む | !Sub app-bucket-${env} | app-bucket-dev |
| !GetAtt | リソースの属性を取得する | !GetAtt AppLambdaRole.Arn | IAM RoleのARN |

👉 `${env}` を文字列展開する場合は `!Sub` を使います。

### 3-2. Parameters( 外部入力値 )
Parameters は、テンプレート内で後から変更できる入力項目です。  
同じYAMLでも、値を変えるだけで別環境や別設定に使い回せます。

```yaml
Parameters:
  env:
    Type: String
    Default: dev
```

- `Type` : 値の種類(文字列、数値など)
- `Default` : 未指定時に使われる初期値

この例では `env` という名前の入力項目を定義しており、現場では開発・検証・本番などの環境名切替に利用します。  
　( stg は staging(検証環境) の略です )

| 環境 | 用途 |
|---|---|
| dev  | 開発・検証 |
| stg  | 本番前確認・総合テスト |
| prod | 本番利用 |

#### 📌 主な利用例
他にも、よく使うParametersの例があります。
- リソース名変更：`bucketName` ( バケット名 ) など
- 性能変更：`instanceType` ( インスタンスサイズ )
- 通知先変更：`notifyEmail` ( メールアドレス )

👉 IaCなら、同じテンプレートに対してパラメータだけ変えて複数環境を作成できます。

#### 📌 実行例
実行時に上書きもできます。  
```bash
aws cloudformation deploy --parameter-overrides env=stg
```

### 3-3. Resources( リソース定義 )
Resources は「何を作るか」と「どう設定するか」を定義する、CloudFormationの中心項目です。  
CloudFormation では、`Type` に作成するAWSリソース、`Properties` に詳細設定を記述します。  
`Properties` の内容はAWSリソースごとに異なり、名前・性能・動作設定などを記述します。  
作成されるリソースは、IAM Role など他リソースと組み合わせて利用されます。

例：S3バケットを定義する場合
```yaml
Resources:

  AppBucket:
    Type: AWS::S3::Bucket
    Properties:
      BucketName: !Sub app-bucket-${env}
      Tags:
        - Key: Env
          Value: !Ref env
```
このS3バケットへ Lambda などがアクセスする場合は、IAM Role による権限設定も必要になります。

| 項目 | 内容 |
|---|---|
| AppBucket | 論理ID |
| Type | 作成する種類 |
| Properties | 詳細設定 |
| BucketName | S3バケット名 |
| Tags | 管理ラベル |

#### 📌 この資料で扱う主な Type 一覧
| 分類 | Type |
|---|---|
| S3 | AWS::S3::Bucket |
| Lambda | AWS::Lambda::Function |
| IAM Role | AWS::IAM::Role |
| DynamoDB | AWS::DynamoDB::Table |
| VPC | AWS::EC2::VPC |
| Subnet | AWS::EC2::Subnet |
| Security Group | AWS::EC2::SecurityGroup |
| ECS Cluster | AWS::ECS::Cluster |
| ECS Service | AWS::ECS::Service |

👉 Resources は CloudFormation の中心であり、他の定義と連携して動作します。

#### 📌 Tag( タグ設計 )
Tag は管理用ラベルとして、実務では非常に重要です。  
AWSリソースの識別、検索、コスト管理などに利用されます。
```
Tags:
  - Key: Project
    Value: ProjectA
  - Key: System
    Value: SalesApp
  - Key: Env
    Value: !Ref env
```
- `Key`：タグの分類名
- `Value`：設定値( 固定値またはパラメータ参照 )

#### 📌 主な用途
| 用途 | 例 |
|---|---|
| コスト集計 | TeamA など |
| 環境識別 | dev / stg / prod |
| システム識別 | SalesApp など |

👉 タグ設計がない運用は、後で管理・集計が困難になります。

### 3-4. Outputs( 出力値 )
Outputs は、作成したAWSリソースの情報を表示したり、他スタックや外部処理から参照しやすくするための項目です。  
作成後に、リソース名やURLなどを確認することもできます。

```yaml
Outputs:
  BucketName:
    Value: !Ref AppBucket
```
- 左側 `BucketName` は表示名
- `Value` に表示したいAWSリソース情報を指定します。

例：
- 作成したS3バケット名
- Lambda ARN
- API URL
- DB接続先Endpoint

👉 Outputs は「作った結果を外に見せる窓口」です。

## 4. IAM Role( 権限設定 )
IAM Roleは、AWSサービスに対して「どのAWS操作を許可するか」を定義する仕組みです。

AWSではセキュリティ上、サービス同士が自由にアクセスできないため、  
IAM Roleを使って明示的に権限を付与します。

例えば Lambda は、単体では S3 にアクセスできません。  
そのため、IAM Roleを通じて「S3にアクセスしてよい」という許可を与えます。

### 📌 例：Lambda → S3アクセス
```plaintext id="iam-flow"
Lambda
  ↓ ( IAM Roleで許可 )
S3へアクセス可能になる
```

### 📌 よくある権限例
- Lambda → S3書き込み
- Lambda → CloudWatch Logs出力
- Firehose → S3保存

### 💡 重要ポイント
- AWSでは「ユーザに権限を付ける」のではなく
  **サービスにRoleを付けるのが基本**
- アクセスキー直書きは避ける
- Roleは最小権限で設計する

👉 IAM Roleは「操作の許可証」のようなものです

## 5. 実行コマンド( 現場で重要 )

### 5-1 構文確認
```bash
aws cloudformation validate-template \
--template-body file://template.yaml
```

### 5-2 変更差分確認( 任意 )
本番環境などでは、反映前に変更内容を確認すると安全です。

```bash
aws cloudformation create-change-set \
--stack-name sample-stack \
--template-body file://template.yaml \
--change-set-name review-001
```

👉 何が作成・変更・削除されるか事前確認できます。

### 5-3 初回作成 / 更新
```bash
aws cloudformation deploy \
--template-file template.yaml \
--stack-name sample-stack \
--capabilities CAPABILITY_NAMED_IAM
```

### 5-4 削除
```bash
aws cloudformation delete-stack \
--stack-name sample-stack
```

### 5-5 学習時によくある削除失敗
CloudFormation で削除しても、内容物や依存関係により  
Stack が正常に削除できない場合があります。

- S3 バケット内にファイルが残っている
- IAM Role が他サービスから参照されている
- Network Interface などネットワーク関連が残っている

👉 削除失敗時は CloudFormation Events を確認し、原因リソースを先に解消してから再実行します。

## 6. 基本作業フロー
```plaintext
① 作りたい構成を整理
② YAML作成
③ validate
④ deploy
⑤ AWS環境確認
⑥ 修正して再deploy
⑦ 不要ならdelete
```

👉 実務では「いきなり書く」より、先に構成整理してから作る方が失敗しにくいです。

## 7. 学習時の注意( 課金と削除 )
CloudFormation自体は無料ですが、
作成されるAWSリソースには料金が発生する場合があります。

例：
- EC2：起動時間で課金
- RDS：起動中課金
- NAT Gateway：高額注意
- S3：低コスト

👉 学習後は delete-stack を忘れず実施します。

## 8. まとめ
- IaCはインフラをコードで管理する方法
- AWSではCloudFormationが代表的
- Resourcesで構成を定義する
- Parametersで環境差分を管理する
- Tag / IAM Role は実務で重要
- validate → deploy → update → delete が基本操作
- IaCはコード作成だけでなく、安全な変更管理と運用まで含む

👉 **IaCは「書くだけ」でなく「実行して運用する」までが本番です。**

## 9. 重要用語説明
| 用語 | 説明 |
|---|---|
| IaC( インフラコード管理 )<br>Infrastructure as Code | インフラ構成をコードで管理し、再現性と自動化を高める手法。 |
| CloudFormation( AWS構成管理 )<br>AWS CloudFormation | AWSリソースをテンプレートで作成・更新・削除できるサービス。 |
| YAML( 設定記述形式 )<br>YAML Ain't Markup Language | 人が読みやすい構成ファイル形式。 |
| Resources( リソース定義 )<br>Resources | 作成対象となるAWSサービスを記述する項目。 |
| Logical ID( 論理ID ) | CloudFormationテンプレート内で、各リソースを識別する名前。 |
| Parameters( 外部入力値 )<br>Parameters | 環境ごとに値を切り替えるための入力変数。 |
| Outputs( 出力値 )<br>Outputs | 作成後にIDやURLなどを表示する出力項目。 |
| Tag( 識別ラベル )<br>Tag | AWSリソースへ付与する管理用ラベル。 |
| IAM Role( 権限ロール )<br>Identity and Access Management Role | AWSサービスへ権限を委任する仕組み。 |
| Stack( 管理単位 )<br>Stack | CloudFormationでまとめて管理する構成単位。 |
| Deploy( 反映 )<br>Deploy | コードや設定をAWS環境へ適用すること。 |
