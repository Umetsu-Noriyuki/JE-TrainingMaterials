# IaC基礎と実践<br>( Infrastructure as Code、CloudFormation、YAML、Deploy )

## 1. IaC( Infrastructure as Code )とは？

IaC( Infrastructure as Code )とは、  
サーバー・ネットワーク・権限設定などのインフラ構成を、**コードとして管理する考え方**です。

従来は管理画面から手作業で構築していましたが、IaCでは設定内容をファイル化し、再現可能な形で管理します。

- **目的**：構築作業の標準化、自動化、再現性向上
- **効果**：設定ミス削減、レビュー可能、変更履歴管理
- **代表例**：CloudFormation、Terraform

```plaintext
[従来]
　管理画面で手作業設定

[IaC]
　コードを書く → 実行 → 同じ環境を再現
```

👉 **インフラを人の記憶ではなく、コードで管理するのがIaCです。**

## 2. AWS標準IaC - CloudFormation
CloudFormation は AWS が提供する IaC サービスです。  
YAML / JSON でAWS構成を定義し、その内容をAWSへ反映できます。

- AWS公式サービスのため相性が良い
- 多くのAWSサービスに対応
- 変更内容の差分( Change Set )を事前確認して安全に変更を適用できる
- 複数のリソースをまとめたスタック( Stack )という単位で管理できる

```plaintext
YAMLファイル
   ↓
CloudFormation
   ↓
VPC / S3 / Lambda / IAM など作成
```

## 3. CloudFormationテンプレート( YAML )について
CloudFormationでは、YAMLファイルにAWS構成を書き、環境を自動作成します。

### 3-1. 基本構造
テンプレートの中に、複数のAWSリソースをまとめて記述できます。  

```plaintext
Root( テンプレート全体 )
├─ AWSTemplateFormatVersion
├─ Parameters( 入力値 )
├─ Resources( 作成するAWSリソース )
│   ├─ 論理ID 1( AWSリソース 1: Type + Properties )
│   ├─ 論理ID 2( AWSリソース 2: Type + Properties )
│   ├─  ...
│   └─ 論理ID N( AWSリソース N: Type + Properties )
└─ Outputs( 作成結果 )
```
👉 CloudFormationは「単一サービスの定義」ではなく「複数リソースの集合」です。  

以下は主要要素を含んだ最小例です。
```yaml
AWSTemplateFormatVersion: '2010-09-09'

Parameters:
  env:
    Type: String
    Default: dev

Resources:

  AppBucket:
    Type: AWS::S3::Bucket
    Properties:
      BucketName: !Sub app-bucket-${env}
      Tags:
        - Key: Env
          Value: !Ref env

  AppLambdaRole:
    Type: AWS::IAM::Role
    Properties:
      RoleName: !Sub app-lambda-role-${env}
      # 利用対象: Lambda
      AssumeRolePolicyDocument:
        Version: '2012-10-17'
        Statement:
          - Effect: Allow
            # 利用対象: Lambda
            Principal:
              Service: lambda.amazonaws.com
            Action: sts:AssumeRole
      # 権限内容: 許可される操作
      Policies:
        # 権限名: S3書き込み
        - PolicyName: S3WritePolicy
          PolicyDocument:
            Version: '2012-10-17'
            Statement:
              - Effect: Allow
                Action:
                  - s3:PutObject
                # 対象リソース: ARN
                Resource: !Sub arn:aws:s3:::app-bucket-${env}/*
      Tags:
        - Key: Env
          Value: !Ref env

  AppLambda:
    Type: AWS::Lambda::Function
    Properties:
      FunctionName: !Sub app-lambda-${env}
      # 実行環境: 使用するランタイム
      Runtime: python3.11
      # 実行関数: 呼び出される関数
      Handler: index.lambda_handler
      # 割り当てRole: ARN
      Role: !GetAtt AppLambdaRole.Arn
      Tags:
        - Key: Env
          Value: !Ref env
      Code:
        ZipFile: |
          # Lambda実行コード
          def lambda_handler(event, context):
              return {"statusCode": 200, "body": "ok"}

Outputs:
  BucketName:
    Value: !Ref AppBucket
```
👉 Parameters は入力値、Resources は作成内容、Outputs は作成結果です。

#### 📌 上記YAMLのリソース対応表
| # | 論理ID | Type | 役割 | 物理ID( env=dev ) |
|---|---|---|---|---|
| 1 | AppBucket | S3 Bucket | データ保存 | app-bucket-dev |
| 2 | AppLambdaRole | IAM Role | Lambdaへ権限付与 | app-lambda-role-dev |
| 3 | AppLambda | Lambda | Pythonコード実行 | app-lambda-dev |

- 論理ID( Logical ID ): テンプレート内で管理する名前
- 物理ID( Physical ID ): AWS上で実際に作成されるリソース名

#### 📌 !Ref / !Sub / !GetAtt とは？
CloudFormation では、他の値を参照したり、文字列に値を埋め込む専用記法があります。

| 記法 | 用途 | 例 | 結果( env=dev ) |
|---|---|---|---|
| !Ref | 値を参照する | !Ref env | dev |
| !Sub | 文字列内に値を埋め込む | !Sub app-bucket-${env} | app-bucket-dev |
| !GetAtt | リソースの属性を取得する | !GetAtt AppLambdaRole.Arn | IAM RoleのARN |

👉 `${env}` を文字列展開する場合は `!Sub` を使います。

### 3-2. Parameters( 外部入力値 )
Parameters は、テンプレート内で後から変更できる入力項目です。  
同じYAMLでも、値を変えるだけで別環境や別設定に使い回せます。

```yaml
Parameters:
  env:
    Type: String
    Default: dev
```

- `Type` : 値の種類(文字列、数値など)
- `Default` : 未指定時に使われる初期値

この例では `env` という名前の入力項目を定義しており、現場では開発・検証・本番などの環境名切替に利用します。  
　( stg は staging(検証環境) の略です )

| 環境 | 用途 |
|---|---|
| dev  | 開発・検証 |
| stg  | 本番前確認・総合テスト |
| prod | 本番利用 |

#### 📌 主な利用例
他にも、よく使うParametersの例があります。
- リソース名変更：`bucketName` ( バケット名 ) など
- 性能変更：`instanceType` ( インスタンスサイズ )
- 通知先変更：`notifyEmail` ( メールアドレス )

👉 IaCなら、同じテンプレートに対してパラメータだけ変えて複数環境を作成できます。

#### 📌 実行例
実行時に上書きもできます。  
```bash
aws cloudformation deploy --parameter-overrides env=stg
```

### 3-3. Resources( リソース定義 )
Resources は「何を作るか」と「どう設定するか」を定義する、CloudFormationの中心項目です。  
CloudFormation では、`Type` に作成するAWSリソース、`Properties` に詳細設定を記述します。  
`Properties` の内容はAWSリソースごとに異なり、名前・性能・動作設定などを記述します。  
作成されるリソースは、IAM Role など他リソースと組み合わせて利用されます。

例：S3バケットを定義する場合
```yaml
Resources:

  AppBucket:
    Type: AWS::S3::Bucket
    Properties:
      BucketName: !Sub app-bucket-${env}
      Tags:
        - Key: Env
          Value: !Ref env
```
このS3バケットへ Lambda などがアクセスする場合は、IAM Role による権限設定も必要になります。

| 項目 | 内容 |
|---|---|
| AppBucket | 論理ID |
| Type | 作成する種類 |
| Properties | 詳細設定 |
| BucketName | S3バケット名 |
| Tags | 管理ラベル |

#### 📌 よく使うType一覧

| 分類 | Type | 主な用途 | 使用章 |
|---|---|---|---|
| VPC | AWS::EC2::VPC | ネットワーク分離 |  |
| Subnet | AWS::EC2::Subnet | VPC内ネットワーク分割 |  |
| Security Group | AWS::EC2::SecurityGroup | 通信制御 |  |
| EC2 | AWS::EC2::Instance | 仮想サーバ | |
| API Gateway | AWS::ApiGateway::RestApi | HTTP API公開 | 3-2.サーバレス |
| EventBridge Rule | AWS::Events::Rule | 定時実行・イベント条件定義 | 3-3.イベント通知 |
| Lambda | AWS::Lambda::Function | プログラム実行 | 3-2.サーバレス<br>3-3.イベント通知 |
| IAM Role | AWS::IAM::Role | AWSサービスへ権限付与 | 3-2.サーバレス<br>3-4.非同期 |
| Lambda Permission | AWS::Lambda::Permission | サービス間の実行許可 | 3-3.イベント通知 |
| Logs | AWS::Logs::LogGroup | ログ保存・保持期間管理 | 3-2.サーバレス |
| SNS Topic | AWS::SNS::Topic | 通知配信 | 3-3.イベント通知 |
| SQS | AWS::SQS::Queue | 非同期メッセージ処理 | 3-4.非同期 |
| S3 | AWS::S3::Bucket | ファイル・データ保存 | 3-2.サーバレス<br>3-4.非同期 |
| CloudFront | AWS::CloudFront::Distribution | CDN配信 | |
| RDS | AWS::RDS::DBInstance | リレーショナルDB | |
| DynamoDB | AWS::DynamoDB::Table | キー・バリュー型データ保存 |  |
| Firehose | AWS::KinesisFirehose::DeliveryStream | データ配送 | 3-4.非同期 |
| Glue | AWS::Glue::Job | データ加工・変換 | 3-4.非同期 |
| ECS Cluster | AWS::ECS::Cluster | コンテナ実行基盤 |  |
| ECS Service | AWS::ECS::Service | コンテナ常駐実行 |  |

👉 Resources は CloudFormation の中心であり、他の定義と連携して動作します。

#### 📌 Tag( タグ設計 )
Tag は管理用ラベルとして、実務では非常に重要です。  
AWSリソースの識別、検索、コスト管理などに利用されます。

※ Tags はリソースごとに設定し、対応リソースのみ利用可能です。
```
Resources:

  {論理ID}:
    Properties:
      Tags:
        - Key: Owner
          Value: ProjectA
        - Key: System
          Value: Sales
        - Key: Env
          Value: !Ref env
```

- `Key`：タグの分類名
- `Value`：設定値( 固定値またはパラメータ参照 )

#### 📌 主な用途
| Key | 用途 | 説明 | 例 |
|---|---|---|---|
| Owner | コスト管理 / 管理責任 | リソースの管理責任者( 誰が管理するか ) | ProjectA / TeamA |
| System | システム識別 | 何のシステムかを識別する | Sales |
| Env | 環境識別 | どの環境かを識別する | dev / stg / prod |

👉 タグはコスト管理や識別に重要で、実務では `Owner` を付けて管理責任者を明確にするのが一般的です。

### 3-4. Outputs( 出力値 )
Outputs は、作成したAWSリソースの情報を表示したり、他スタックや外部処理から参照しやすくするための項目です。  
作成後に、リソース名やURLなどを確認することもできます。

```yaml
Outputs:
  BucketName:
    Value: !Ref AppBucket
```
- 左側 `BucketName` は表示名
- `Value` に表示したいAWSリソース情報を指定します。

例：
- 作成したS3バケット名
- Lambda ARN
- API URL
- DB接続先Endpoint

👉 Outputs は「作った結果を外に見せる窓口」です。

## 4. IAM Role( 権限設定 )
CloudFormationでAWSリソースを作成する際は、リソース作成だけでなくアクセス権限の設定も重要です。

IAM Roleは、AWSサービスに対して「どのAWS操作を許可するか」を定義する仕組みです。

AWSではセキュリティ上、サービス同士が自由にアクセスできないため、  
IAM Roleを使って明示的に権限を付与します。

例えば Lambda は、初期状態では S3 にアクセスできません。  
そのため、IAM Roleを通じて「S3にアクセスしてよい」という許可を与えます。

例：CloudFormation で IAM Role を定義する場合
```yaml
Resources:

  AppLambdaRole:
    Type: AWS::IAM::Role
    Properties:
      RoleName: !Sub app-lambda-role-${env}
      AssumeRolePolicyDocument:
        Version: '2012-10-17'
        Statement:
          - Effect: Allow
            Principal:
              Service: lambda.amazonaws.com
            Action: sts:AssumeRole
      Policies:
        - PolicyName: S3WritePolicy
          PolicyDocument:
            Version: '2012-10-17'
            Statement:
              - Effect: Allow
                Action:
                  - s3:GetObject
                  - s3:PutObject
                  - s3:DeleteObject
                Resource: !Sub arn:aws:s3:::app-bucket-${env}/*
```
※ `AssumeRolePolicyDocument` は「誰がこのRoleを使えるか」を定義します。  
実際に許可する操作( S3書込など )は `Policies` などで別途定義します。

### 📌 例：Lambda → S3アクセス
```
Lambda
  ↓ IAM Role を利用
  ↓ S3書込権限を持つ
S3へアクセス可能
```

### 📌 よくある権限例
- Lambda → S3書き込み
- Lambda → CloudWatch Logs出力
- Firehose → S3保存

### 💡 重要ポイント
- CloudFormationやLambda連携では、**サービスにRoleを付けるのが基本**
- アクセスキー直書きは避ける
- Roleは最小権限で設計する

👉 IAM Roleは「操作の許可証」のようなものです

### 📌 よく使うAction一覧
IAM Policy の `Action` には、サービスごとに様々な操作があります。  
ここでは、実務でよく使う代表的なものをまとめます。

| サービス | Action | 内容 | 用途 | 使用章 |
|---|---|---|---|---|
| Lambda | lambda:InvokeFunction | Lambda実行 | 実行 | 3-2.サーバレス<br>3-3.イベント通知 |
| CloudWatch Logs | logs:CreateLogGroup | ロググループ作成 | 初期化 | 3-2.サーバレス |
| CloudWatch Logs | logs:CreateLogStream | ログストリーム作成 | 初期化 | 3-2.サーバレス |
| CloudWatch Logs | logs:PutLogEvents | ログ出力 | ログ記録 | 3-2.サーバレス |
| SQS | sqs:SendMessage | メッセージ送信 | 非同期投入 | 3-4.非同期 |
| SQS | sqs:ReceiveMessage | メッセージ受信 | 非同期処理 | 3-4.非同期 |
| SQS | sqs:DeleteMessage | メッセージ削除 | 完了処理 | 3-4.非同期 |
| S3 | s3:GetObject | ファイル取得 | 読み込み | 3-4.非同期 |
| S3 | s3:PutObject | ファイル作成・更新 | 書き込み | 3-4.非同期 |
| S3 | s3:DeleteObject | ファイル削除 | 削除 |  |
| S3 | s3:ListBucket | バケット内一覧取得 | 一覧確認 |  |
| DynamoDB | dynamodb:GetItem | データ取得 | 読み込み |  |
| DynamoDB | dynamodb:PutItem | データ登録 | 書き込み |  |
| DynamoDB | dynamodb:UpdateItem | データ更新 | 更新 |  |
| DynamoDB | dynamodb:DeleteItem | データ削除 | 削除 |  |
| Firehose | firehose:PutRecord | データ送信 | 配送 | 3-4.非同期 |
| Glue | glue:StartJobRun | Glueジョブ実行 | ETL開始 | 3-4.非同期 |
| SNS | sns:Publish | SNS通知送信 | 通知 | 3-3.イベント通知 |

👉 実務では「必要な操作だけを許可する( 最小権限 )」が基本です

## 5. 実行コマンド( 現場で重要 )
AWS CLIを使用して CloudFormation テンプレートを操作します。  
基本的な流れは「確認 → 作成 → 更新 → 削除」です。

aws cloudformation には主に以下のサブコマンドがあります。

| サブコマンド | 説明 |
|---|---|
| validate-template | YAML構文確認 |
| create-change-set | 変更差分( Change Set )の事前確認 |
| deploy | スタックの作成・更新 |
| delete-stack | スタック削除 |

👉 本番環境では変更差分確認( create-change-set )を行うことで安全性が高まります

### 5-1 コマンド実行例
CloudFormation の基本操作コマンド例です。

```bash
# YAML構文確認
aws cloudformation validate-template \
--template-body file://template.yaml

# 変更差分の事前確認
aws cloudformation create-change-set \
--stack-name sample-stack \
--template-body file://template.yaml \
--change-set-name review-001

# スタックの作成・更新
aws cloudformation deploy \
--template-file template.yaml \
--stack-name sample-stack \
--capabilities CAPABILITY_NAMED_IAM

# スタック削除
aws cloudformation delete-stack \
--stack-name sample-stack
```

👉 基本操作は「確認 → 作成/更新 → 削除」の流れです。

### 5-2 スタック削除にて よくある失敗事例
CloudFormation で削除しても、内容物や依存関係により  
Stack が正常に削除できない場合があります。

- S3 バケット内にファイルが残っている
- IAM Role が他サービスから参照されている
- Network Interface などネットワーク関連が残っている

👉 削除失敗時は CloudFormation Events を確認し、原因リソースを先に解消してから再実行します。

## 6. 基本作業フロー

### 🧭 IaCの流れ

```plaintext
[ 設計 ]
   ↓
[ 作成( YAML ) ]
   ↓
[ validate ]
   ↓
[ deploy ]
   ↓
[ AWS環境確認 ]
   ↓
[ エラー修正 → 再deploy ]
   ↓
[ delete ]
```

👉 実務では「いきなり書く」より、先に構成整理してから作る方が失敗しにくいです。

## 7. 学習時の注意( 課金と削除 )
CloudFormation自体は無料ですが、
作成されるAWSリソースには料金が発生する場合があります。

例：
- EC2：起動時間で課金
- RDS：起動中課金
- NAT Gateway：高額注意
- S3：低コスト

👉 学習後は delete-stack を忘れず実施します。

### 💰 課金の目安
| サービス | 課金 | 備考 |
|---|---|---|
| S3 | 安い | 学習向け |
| Lambda | ほぼ無料 | 無料枠あり |
| EC2 | 有料 | 起動時間課金 |
| NAT Gateway | 高額 | 学習では避ける |

### ⚠ よくある課金事故
- NAT Gatewayを作成したまま放置 → 数千円/日
- RDSを削除せず放置 → 数千円/月
- ALB + ECS構成 → 常時課金

### 🛡 学習ルール( 推奨 )
- 必ず `delete-stack` までを1セットで実施
- 高額リソース( NAT / RDS )は使用しない

👉 削除できない場合は「5-2 よくある失敗事例」を参照

## 8. まとめ
👉 **IaCとは「インフラをコードで再現できる状態にすること」**

### 📌 要点整理
- IaCはインフラをコードで管理する方法
- AWSではCloudFormationが代表的
- Resourcesで構成を定義する
- Parametersで環境差分を管理する
- Tag / IAM Role は実務で重要
- validate → deploy → update → delete が基本操作

### ❌ よくある誤解
- IaC = YAMLを書くこと → ❌
- IaC = 自動化ツール → ❌

👉 **IaCはツールではなく「考え方」**

### 💼 実務での使われ方
- Gitでレビューされる
- CI/CDで自動deployされる
- 本番環境の変更はIaC経由で実施する

### ⚠ IaCがない場合
- 手作業ミスが発生
- 環境の再現ができない
- 属人化する

### ✅ IaCがある場合
- 同じ環境を再現可能
- 自動化できる
- 変更履歴が残る

👉 **IaCは「書くこと」が目的ではなく「安全に運用するための仕組み」です**
